## 1. 文書概要

### 1.1 文書名

AI/LLMエンジニア向けポートフォリオ提出用 RAGシステム 要件定義書

### 1.2 目的

本書は、**AI/LLMエンジニアが企業へ提出するポートフォリオ作品としてのRAGシステム** に関する要件を定義する。

本システムは、単なる転職支援ツールではなく、以下を採用側に示すための成果物とする。

- RAGアーキテクチャの理解
    
- LLMアプリケーション実装力
    
- 情報検索設計力
    
- セキュリティ設計力
    
- 評価設計力
    
- 運用・保守設計力
    
- クロスOS、CI/CD、コンテナ運用への理解
    

### 1.3 背景

本システムの主眼は、**採用選考で提示できる完成度の高いRAGシステムを構築すること** にある。  
画面で見たときの完成度、回答精度担保、出典明記、信頼度表示、会話の継続性、評価と改善の仕組みを重視する。面接等でのデモ時間はおおむね5分を想定し、短時間でも設計意図と品質の高さが伝わる構成を採る。

## 5. 開発・運用前提

### 5.1 開発環境

- 主開発環境は **Windows**
    
- Windowsネイティブを基本とする
    
- Docker Desktop を前提とする
    
- 補助的に WSL2 が利用できることは許容する
    
- Python実行は原則コンテナで統一する
    
- パッケージ管理補助として `uv` の採用を許容する
    

### 5.2 運用環境

- 主運用環境は **Ubuntu 24.04.4 LTS**
    
- Docker Compose による運用を前提とする
    
- systemd によるアプリ直管理は行わない
    
- Phase1 の deploy 先は **ローカル検証環境**
    
- Phase3 以降は **AWS環境へのdeploy** を対象に含める
    
- Ubuntu環境ではGPUなしも想定し、外部API利用に切り替え可能な設計とする
    

### 5.3 クロスOS要件

- Windows と Ubuntu の両方で主要機能が動作すること
    
- パス区切り、改行コード、ファイル監視挙動、権限差異、大文字小文字差異、シェル差異に配慮すること
    
- コンテナにより実行環境差分を吸収すること
    
- ドキュメント上でOS別起動手順を明記すること

## 7. 技術選定方針

### 7.1 FastAPI

RAG、LLM、評価、バッチ、APIをPythonで一貫して実装しやすく、型ヒント・バリデーション・OpenAPI生成・非同期対応の相性が良いため採用する。

### 7.2 React

ChatGPTライクなチャット体験、会話一覧、設定画面、管理画面、引用パネルなどをコンポーネントベースで構成しやすく、API分離構成とも相性が良いため採用する。

### 7.3 SQLAlchemy 2.x

永続化、会話履歴、監査ログ、評価結果、ユーザー権限、文書メタデータ等のRDB管理において、Pythonエコシステムとの相性が良く、保守性と実務性のバランスが良いため採用する。

### 7.4 Pydantic v2

API request / response のスキーマ、内部DTO、設定スキーマ、検証ロジックの一貫性を保ちやすいため採用する。

### 7.5 Alembic

DBスキーマ変更履歴を管理し、マイグレーションを再現性高く運用するため採用する。

### 7.6 BAAI / bge-m3

多言語対応、長文対応、将来の hybrid retrieval への拡張性、RAG用途での説明しやすさのバランスが良いため採用する。

### 7.7 BAAI / bge-reranker-v2-m3

多言語対応、実務的な使いやすさ、Phase1から入れやすい性能と構成のバランスが良いため採用する。

### 7.8 PaddleOCR

将来のOCR・構造化文書解析・マルチモーダル拡張に自然につながり、ポートフォリオとして説明しやすいため採用する。

### 7.9 Docker

Windows開発環境とUbuntu運用環境の差異を吸収し、再現性の高い起動方法を提供するため採用する。

### 7.10 GitHub Actions

lint、テスト、ビルド、イメージ作成、検証環境向けデプロイまで自動化し、継続的品質保証を示すため採用する。

## 10. 会話機能

### 10.1 会話セッション

- 会話はセッション単位で保存する
    
- 各セッションはタイトルを持てる
    
- タイトルは UI / CLI / API から変更できる
    
- タグ付けができる
    
- ピン留めできる
    
- 削除できる
    
- 一時チャットを作成できる
    

### 10.2 一時チャット

- UIから離れたら履歴一覧に表示しない
    
- サーバー側 TTL により自動削除する
    
- 初期 TTL は **120分**
    
- 一時チャットは恒久保存を前提としない
    
- 少なくとも通常履歴管理とは分離して扱う
    
- 実装上は `chat_sessions` の属性として管理してもよい
    

### 10.3 会話メモリ

- 会話メモリは **要約メモリ + 直近8件** を基本とする
    
- 直近参照件数 `N` は設定可能とし、初期値は `8`
    
- 会話メモリはセッション単位で管理する
    
- 会話メモリは retrieval の検索対象には含めない
    
- 会話メモリはユーザー意図解釈や文脈補助のために利用する
    

### 10.4 会話再開

- 保存済み会話を選択して再開できること
    
- 再開時には当該セッションの要約メモリと直近メッセージを参照できること
    

### 10.5 メッセージ編集

- メッセージ編集は **ChatGPTと同様の仕様** を目標とする
    
- 過去メッセージ編集時は、その地点以降の会話系列を再生成対象とする
    
- 旧系列は内部的に保持する
    
- 明示的な分岐UIは必須としない
    
- UI上は自然な連続会話として扱えること
    

### 10.6 会話保存項目

- session_id
    
- title
    
- tags
    
- pinned
    
- temporary_flag
    
- ttl_expires_at
    
- created_at
    
- updated_at
    
- message role
    
- message body
    
- message edited_flag
    
- message lineage / internal branch reference
    
- summary memory
    
- linked retrieval run
    
- linked citations

### 11.3 文書状態

文書は以下の状態遷移を持つ。

- uploaded
    
- queued
    
- parsing
    
- parsed
    
- chunking
    
- embedding
    
- indexing
    
- ready
    
- failed
    
- pending_review
    
- archived
    

### 11.4 バージョン管理

- 文書は **logical_documents + document_versions** の二層で管理する
    
- `documents` という独立概念は持たない
    
- active version を管理する
    
- 新版登録時は新バージョンを作成する
    
- 旧版も保持する
    
- 旧版検索を許容する
    
- active version を優先する
    
- 旧版採用時は応答および citation に明示表示する
    
- 通常の削除運用はアーカイブを標準とし、物理削除は管理者限定とする
    

### 11.5 差分

- 差分表示は Phase2
    
- 将来的に「このバージョンで変更がありました」を表現できる構造にする
    

### 11.6 取り込み経路

- GUI Upload
    
- CLI
    
- API
    
- Folder Watch
    

### 11.7 アップロード防御

- 拡張子検証
    
- MIME type 検証
    
- ファイルサイズ上限チェック
    
- 対応形式以外の拒否
    
- 危険ファイル拒否方針
    
- 圧縮ファイルや将来のアーカイブ入力に対する圧縮爆弾対策に拡張できること

- 実装フェーズは **Phase3**
    
- OCR エンジンは **PaddleOCR**
    
- OCR結果は将来的に citation / chunk metadata / confidence と紐づけられる構造にする
    

### 13.3 チャンク分割

- 見出し境界を優先する
    
- 長文段落は追加分割できる
    
- overlap は設定可能とする
    
- chunk size は設定可能とする
    
- 初期値は **512 tokens / overlap 128 tokens**
    
- 表は別種のチャンクとして扱えることが望ましい
    
- parent-child chunk 構造は Phase2 以降で強化する

## 15. 回答生成

### 15.1 基本方針

- 回答は retrieval 結果に基づいて生成する
    
- 根拠のない断定を避ける
    
- 引用のない主張を禁止する
    
- 非公開ソースを使った場合はその旨を表示する
    

### 15.2 citation

- 参照文書名を表示できること
    
- 該当箇所の引用を表示できること
    
- 公開情報ならURLを表示できること
    
- 非公開ならその旨を明示できること
    
- citation は少なくとも以下を保持できること
    
    - citation_id
        
    - retrieval_run_id
        
    - document_version_id
        
    - chunk_id
        
    - snippet
        
    - page_from / page_to
        
    - source_type
        
    - source_url
        
    - display_label
        
    - rank_order
        

### 15.3 信頼度

- 信頼度表示を行う
    
- 信頼度しきい値は設定可能とする
    
- retrieval score と answer confidence の両方を分離して扱える設計とする
    
- 少なくとも以下を内部的に扱えること
    
    - retrieval_score_summary
        
    - rerank_score_top1
        
    - answer_confidence
        
    - groundedness_score
        
    - confidence_label
        
- confidence_label は少なくとも `High / Medium / Low` の3段階で表示できること
    
- UI と API の両方で信頼度情報を返却できること
    

### 15.4 曖昧質問

- 曖昧な質問は、一度暫定解釈で検索したうえで、解釈内容をユーザーへ確認できること
    

### 15.5 回答拒否

- 根拠が不十分な場合は回答拒否または確認に移行すること
    
- しきい値は設定可能とし、固定値にベタ埋めしないこと

### 16.1 基本方針

- 将来的に外部API利用版も用意できるようにする
    

### 16.2 送信境界

- 外部APIへ送るのは **retrieval後の必要最小限コンテキストのみ**
    
- 文書全文送信は禁止
    
- 送信前にPII masking を適用する
    
- API利用履歴を監査ログへ残す
    

### 16.3 切替

- ローカルLLM利用時と外部API利用時で、上位レイヤのインターフェースを揃える
    
- モデル差し替え時にUIや保存形式が大きく変わらないこと

## 17. セキュリティ・PII

### 17.1 基本要件

- 認証必須
    
- パスワード安全管理
    
- セッション安全管理
    
- 文書アクセス権限の制御
    
- 監査ログ保存
    
- アップロードファイル検証
    
- prompt injection 対策
    
- ログ出力時の機密情報抑制
    

### 17.2 PII方針

- 一般的な個人識別情報はPIIとして扱う
    
- ログ保存前にマスキングを行う
    
- 外部API送信前にもマスキングを行う
    
- UI表示でも権限に応じてマスキングを行う
    
- ユーザー本人が明示操作した場合にのみ一部情報を開示できる設計が望ましい
    
- マスク解除操作は監査対象とすることが望ましい
    

### 17.3 保存ポリシー

- 原文保存: **必要最小限、暗号化、厳格ACL**
    
- ログ / 監査 / 評価: **不可逆マスキング or tokenized ID のみ**
    
- 外部API送信: **不可逆マスク後の最小コンテキスト**
    
- 再識別が必要な項目だけ: **別の secure mapping store で reversible tokenization**
    
- 原文保存とマスク文保存の方針はデータ種別ごとに分離して扱う
    
- 少なくともログ用途・外部送信用のマスクは不可逆を基本とする
    

### 17.4 保持期間

- アプリログ: **30〜90日**
    
- 監査ログ: **180〜365日**
    
- 一時チャット: **TTL 削除**
    
- 原本: **業務上必要な期間のみ**

## 24. 受け入れ基準

### 24.1 システム全体

- 認証なしでは利用できない
    
- 文書アップロードから回答生成まで成立する
    
- 回答に根拠が表示される
    
- 根拠不足時に確認または拒否できる
    
- 会話履歴が保存される
    
- 会話再開時に要約メモリ + 直近8件を参照できる
    
- 会話タイトル変更、タグ、ピン留め、削除ができる
    
- 一時チャットが通常履歴と分離され、TTLで削除される
    
- 文書更新時に pending_review を経て再取り込みできる
    
- active優先、旧版候補可、旧版採用時明示が成立する
    
- chunk 初期値 512 / 128 が設定されている
    
- 埋め込みモデル bge-m3 が適用されている
    
- reranker bge-reranker-v2-m3 が適用されている
    
- 信頼度情報が API および UI で扱える
    
- 外部API送信時に最小限コンテキストのみ送る
    
- GitHub Actions で build / test / deploy が動く
    
- Windows 開発、Ubuntu 24.04.4 LTS 運用で主要機能が再現できる
    

### 24.2 ポートフォリオ観点

- 5分程度で主要価値をデモできる
    
- 技術選定理由を説明できる
    
- セキュリティ配慮を説明できる
    
- 会話機能の完成度を示せる
    
- 評価と改善の仕組みを示せる

## 25. 補足

本システムは、**「転職を助けるRAG」ではなく、「AI/LLMエンジニアとしての能力を企業に示すRAGポートフォリオ」** である。  
したがって、機能選定・評価設計・セキュリティ要件・運用観点は、エンドユーザー向け利便性だけではなく、**採用側に対する説明可能性と技術的説得力** を重視して設計する。

以上をもって、本書を要件定義書 v1.1 とする。

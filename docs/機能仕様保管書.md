# AI/LLMエンジニア向けポートフォリオ提出用 RAGシステム

## 機能仕様補完書 v1.7（最終修正版）

---

## 1. 文書概要

### 1.1 目的

本書は、以下の設計成果物だけでは読み取りづらい **業務ルール / UI連動 / 非同期処理後の見え方 / 実装解釈の統一事項** を補完することを目的とする。

* 基本設計書 v1.2
* ER図 / テーブル設計書 v1.4
* API設計書 v1.6
* 状態遷移仕様書 v1.1
* DDL草案 v1.5

### 1.2 本書の位置づけ

本書は API 契約や DB 定義を上書きしない。
不足する業務ルールと UI 判断基準のみを定義する。

---

# 2. 共通原則

## 2.1 Deterministic 原則

同一条件では同一挙動とする。
実装者判断に依存する optional wording を避ける。

## 2.2 UI 表示原則

内部状態とユーザー向け表示は一致させる必要はない。
ただし誤解を招かないこと。

## 2.3 非同期原則

ジョブ完了前と完了後で UI 状態が変わる場合、本書に従う。

## 2.4 用語整理

* DB テーブル名としての回答根拠: `citations`
* UI 表示上の回答根拠: citation / citation panel
* 回答根拠一覧全体: citations

---

# 3. `/rag/ask` 補完仕様

---

## 3.1 成功時保存単位

成功時は以下を保存する。

* user message
* assistant message
* retrieval_runs
* retrieval_run_items
* citations
* confidence 情報

### 監査・ログ対象

`/rag/ask` 成功時に、通常成功すべてを必ず `audit_logs` へ保存するとは限らない。
以下は監査・ログ対象とする。

* request_id
* external API 利用ログ
* エラー
* 管理操作
* その他、監査対象イベントに該当する操作

`audit_logs` は、外部 API 利用・エラー・管理操作など監査対象イベントに応じて記録する。

---

## 3.2 retrieval 開始前に終了するもの

以下では retrieval_runs を作成しない。

* 入力不正
* 必須項目不足
* 認可失敗
* archived session 投稿禁止
* `request_in_progress` などの duplicate / conflict

`request_in_progress` は失敗ではなく、重複・競合系の応答として扱う。

---

## 3.3 retrieval 開始後失敗

以下では retrieval_runs を保存する。

* Vector DB 一時障害
* rerank 失敗
* generation 失敗
* timeout
* 外部依存障害

---

## 3.4 失敗時 UI 表示

保存されるもの：

* user message

保存されないもの：

* assistant failed placeholder

### 初回送信直後 UI

* user message 表示
* エラー toast / inline error 表示可

### 再読み込み後

* user message のみ表示
* 失敗バナー等は保持しない

---

## 3.5 HTTP ステータス補完

### 422

* 入力不正
* 必須項目不足
* archived session への投稿
* 業務条件不成立

### 403

本書全体の権限制御方針に反する操作。
例：

* admin 専用 endpoint への viewer アクセス
* 管理画面限定操作

### 503

* 外部依存障害
* 一時的処理不能
* timeout

---

## 3.6 Replay 判定

replay 判定は **API レスポンスの `meta.replayed = true` を正とする。**

内部実装で別フラグを持つ場合でも、クライアント判定基準は `meta.replayed` とする。

---

## 3.7 Replay UI

replay 成功時は **通常回答として扱う。**

行わないもの：

* 「再送成功」特別表示
* 再生成成功表示
* 強い通知演出

必要なら開発用 UI のみ replay 表示可。

---

# 4. `/rag/search` 補完仕様

---

## 4.1 保存方針

`/rag/search` は standalone retrieval run として保存する。

* retrieval_runs
* retrieval_run_items
* 必要なログ情報

chat history には表示しない。

---

## 4.2 citations テーブル

`/rag/search` では `citations` テーブルは作成しない。

---

## 4.3 レスポンス表示情報

常に以下を返す。

* snippet
* source label
* page 情報

### page 情報

取得不能時は `null` 許容。

### source label

少なくとも文書名を含むこと。
以下の場合は版識別可能な表示名とする。

* 複数版が候補に混在する場合
* 旧版が候補に含まれる場合

例：

* handbook.pdf
* handbook.pdf (v2)
* policy_manual rev3

---

## 4.4 debug view

standalone retrieval debug は **admin 限定** とし、管理画面配下に配置する。
通常チャット導線とは分離する。

---

# 5. edit / lineage 補完仕様

---

## 5.1 編集対象

編集対象 user message 以降を再生成対象とする。

---

## 5.2 即時反映

編集後の user message 本文は即時表示してよい。

---

## 5.3 job 完了前表示

edit job 完了前は以下が同居しうる。

* 更新済み user message
* 旧 downstream messages

そのため UI は以下のいずれかを出す。

* 再生成中バナー
* loading state
* updating badge

---

## 5.4 表示切替タイミング

新 lineage を通常表示へ切り替えるのは **edit job 成功後** とする。

---

## 5.5 失敗時

edit job 失敗時：

* user message 更新済み
* 旧 lineage 維持
* 新 lineage 未生成

旧 lineage は壊さない。

---

# 6. temporary chat 補完仕様

---

## 6.1 一覧表示

temporary chat は恒久履歴一覧に含めない。

---

## 6.2 セッション中表示

同一 UI セッション中は一時的に表示してよい。

---

## 6.3 TTL

初期 TTL は 120 分。

---

## 6.4 削除対象

TTL 到達後、以下を削除対象とする。

* chat_sessions
* chat_messages
* summary_memories
* chat_tags
* retrieval_runs
* retrieval_run_items
* citations

---

## 6.5 監査ログ / jobs

temporary chat 削除は会話・検索トレース系に限定する。
**audit_logs は削除対象に含めない。**

理由：

* 監査整合性維持
* 操作追跡性確保

jobs については、temporary chat に紐づく会話・検索トレース自体は削除対象としつつ、運用・監査上必要な job record は残してよい。

---

# 7. archive / no-op 補完仕様

---

## 7.1 chat archive

既に archived の場合は no-op success 可。

UI 表示：

* 通常 UI / 管理画面ともに
  「既にアーカイブ済み」

---

## 7.2 document archive

既に archived の場合は no-op success 可。

UI 表示（管理画面標準）：

* 既にアーカイブ済み

archive 後は retrieval 対象外とする。

---

## 7.3 no-op 表示の違い

chat archive は viewer / admin の通常 UI と管理画面の両方で発生しうる。
document archive は admin による管理画面操作を前提とする。
そのため no-op 表示の対象 UI は操作主体に応じて異なる。

---

# 8. 文書版管理 補完仕様

---

## 8.1 active version 優先

検索は active version 優先。

---

## 8.2 旧版候補許可

旧版も候補に含めてよい。
ただし、旧版を候補に含める場合は、回答・citation・UI で旧版利用を明示する。

---

## 8.3 旧版表示条件

以下を満たす場合、通常 UI でも旧版利用表示を行う。

**回答 citations の少なくとも1件が active version ではない document_version を指す場合**

例：

* 旧版ソースを含みます
* 過去版文書を参照しています

---

# 9. jobs / retry / reclaim 補完仕様

---

## 9.1 retry

retry は新 job を作成する。

* retry_of_job_id を持つ
* lineage 増加あり

---

## 9.2 reclaim

reclaim は既存 running job を引き継ぐ処理。

* 新 job 作成なし
* retry lineage 増加なし

---

# 10. 評価機能 補完仕様

---

## 10.1 cancel

Phase1 公開 API に cancel endpoint は持たない。

### canceled 状態用途

* 内部停止
* 将来管理操作予約状態

---

# 11. 権限制御 補完仕様

---

## 11.1 viewer

許可：

* チャット利用
* 通常会話履歴利用
* 自身設定変更

---

## 11.2 admin

追加許可：

* 文書管理
* 評価実行
* 監査ログ閲覧
* debug view 利用
* system_settings 更新

---

## 11.3 trace ID 表示

viewer には通常 UI で以下を表示しない。

* linked_retrieval_run_id
* retrieval_run_id
* internal job_id
* trace token

trace 可視化は admin の debug view 導線に限定する。

---

# 12. 監査ログ 補完仕様

---

## 12.1 外部/API起点イベント

外部/API起点では以下を最低限必須とする。

* request_id
* actor_user_id
* target_type
* target_id
* action_type
* created_at

---

## 12.2 内部イベント

内部 job 起点イベントでは **system-generated request_id** を採番し、必須相当とする。

---

## 12.3 条件付き追加

必要時：

* retrieval_run_id
* job_id
* error_code
* chat_session_id
* logical_document_id
* document_version_id

---

# 13. 実装者向け最終補足

## 13.1 優先順位

実装時は以下の順で整合を確認する。

1. API 契約
2. 状態遷移仕様
3. DDL 制約
4. 本書の UI / 業務ルール補完

本書は API / DB を上書きせず、UI・業務ルールの補完として扱う。

---

## 13.2 代表ルール

* replay 成功時は新 message を作らない
* replay UI は通常回答として扱う
* failed ask でも user message は残す
* retrieval 開始前失敗では retrieval_runs を作らない
* retrieval 開始後失敗では retrieval_runs を残す
* request_in_progress は失敗ではなく duplicate / conflict 系として扱う
* edit 失敗時も旧 lineage は壊さない
* temporary chat は TTL 後に会話・検索トレース系を削除する
* temporary chat 削除でも audit_logs は残す
* retry は元 job を戻さない
* reclaim は既存 job を引き継ぎ、retry lineage を増やさない

---

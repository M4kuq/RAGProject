## 4. FastAPI アプリケーション構成

### 4.1 app 起動

`app/main.py` で FastAPI app を生成する。

主な処理:

- settings 読み込み
    
- middleware 登録
    
- exception handler 登録
    
- router 登録
    
- startup / shutdown hook 登録
    

### 4.2 router prefix

すべての API は `/api/v1` 配下に置く。

### 4.3 middleware order

middleware は以下の順序を基本とする。

1. request_id middleware
    
2. security headers middleware
    
3. CORS middleware
    
4. session middleware
    
5. access log middleware
    

CSRF 検証を middleware として実装する場合は session middleware 後に配置する。  
ただし Phase1 では、CSRF は dependency / service helper として state-changing API で検証してもよい。

### 4.4 request_id

すべての request に `request_id` を付与する。

方針:

- request header `X-Request-Id` があれば検証後に採用
    
- なければ backend で生成
    
- response header `X-Request-Id` に設定
    
- error response の `meta.request_id` にも含める
    
- audit log / application log にも含める
    

### 4.5 X-Request-Id validation

`X-Request-Id` は外部入力であるため、採用前に検証する。

検証条件:

- 最大長を 128 文字とする
    
- 使用可能文字は英数字、`-`、`_`、`.` のみに制限する
    
- 制御文字、改行、空白のみの値は禁止する
    
- 不正な `X-Request-Id` は採用せず、backend で新規生成する
    

目的:

- log injection 防止
    
- audit log 汚染防止
    
- trace ID の可観測性維持

## 6. Database / SQLAlchemy 設計

### 6.1 SQLAlchemy 方針

- SQLAlchemy 2.x style を採用する
    
- `Mapped[]` / `mapped_column()` を使用する
    
- DB access は repository に閉じ込める
    
- session は FastAPI dependency で request ごとに注入する
    

### 6.2 session 管理

`get_db_session()` dependency を用意する。

方針:

- request ごとに session を生成
    
- 正常終了時 commit は service 層で明示する
    
- 例外時 rollback
    
- session close は dependency 側で行う
    

### 6.3 application session 管理

Cookie session は server-side session table を持つ前提とする。

責務:

- session の作成・検証・失効は `core/sessions.py` が担当する
    
- DB session table への access は `SessionRepository` を経由する
    
- AuthService は `core/sessions.py` と `SessionRepository` を用いて login / logout を実行する
    

### 6.4 transaction boundary

transaction は service 層で管理する。

原則:

- 1 API request = 1 transaction を基本とする
    
- 外部 API 呼び出しを DB transaction 内に長時間保持しない
    
- RAG 生成など時間がかかる処理では、必要な DB 保存単位ごとに transaction を分ける
    
- job lease / reclaim は短い transaction で確定する
    

### 6.5 repository 方針

repository は DB 操作のみを担当する。

許可:

- select / insert / update / delete
    
- row lock
    
- pagination query
    
- aggregate query
    

禁止:

- HTTP status の判断
    
- Pydantic response の組み立て
    
- 権限判定
    
- 外部 API 呼び出し
    

### 6.6 IntegrityError 変換方針

DB 制約違反は repository または service 境界で捕捉し、domain error に変換する。

例:

- `client_message_id` partial unique violation -> replay / request_in_progress / conflict 判定へ進む
    
- active retry partial unique violation -> `job_active_retry_exists` / 409
    
- active version partial unique violation -> `active_version_conflict` / 409

## 8. 共通 API Response / Error 設計

### 8.1 成功 response

基本形:

```json
{
  "data": {},
  "meta": {
    "request_id": "..."
  }
}
```

pagination あり:

```json
{
  "data": [],
  "meta": {
    "request_id": "...",
    "pagination": {
      "page": 1,
      "page_size": 20,
      "total": 100,
      "has_next": true
    }
  }
}
```

### 8.2 error response

基本形:

```json
{
  "error": {
    "code": "error_code",
    "message": "user safe message",
    "details": {}
  },
  "meta": {
    "request_id": "..."
  }
}
```

### 8.3 error_code 方針

`core/errors.py` で domain error を定義し、exception handler で HTTP error に変換する。

例:

- `auth_invalid_credentials`
    
- `auth_required`
    
- `permission_denied`
    
- `csrf_invalid`
    
- `csrf_missing`
    
- `resource_not_found`
    
- `validation_error`
    
- `request_in_progress`
    
- `client_message_conflict`
    
- `archived_session_readonly`
    
- `job_active_retry_exists`
    
- `external_dependency_unavailable`
    

`duplicate_content_skipped` は error_code ではなく、成功系 result status として扱う。

archive no-op も error ではなく、成功系 result status として扱う。既に archived の場合は `data.status = already_archived` を返し、`document_already_archived` のような error_code は使用しない。

### 8.4 success result status

成功系だが通常処理とは異なる結果は `data.status` または `result_code` で表現する。

例:

- `duplicate_content_skipped`
    
- `already_archived`
    
- `no_op`
    

### 8.5 exception 変換

|Domain Error|HTTP status|
|---|---|
|AuthenticationRequired|401|
|PermissionDenied|403|
|ResourceNotFound|404|
|ConflictError|409|
|BusinessValidationError|422|
|ExternalDependencyUnavailable|503|

### 8.6 409 / 422 の使い分け

- request structure / 入力値不正: 422
    
- 現在のリソース状態による衝突: 409
    

例:

- `request_in_progress`: 409
    
- `client_message_conflict`: 409
    
- 参照済み文書の physical delete: 409
    
- archived session への投稿: 409
    

archived session への投稿は、入力形式不正ではなく現在のリソース状態による操作不可として扱う。

```text
HTTP status = 409
error.code = archived_session_readonly
```

### 8.7 404 / 403 使い分け

- admin 専用 endpoint に viewer がアクセスした場合は 403
    
- owner 不一致でリソース存在を隠すべき場合は 404
    
- nested path の親子不整合は 404
    
- 認証済みだが操作権限がない管理操作は 403
    

例:

- viewer が admin endpoint にアクセス: 403
    
- 他人の `chat_session_id` を指定: 404
    
- `logical_document_id` 配下に存在しない `document_version_id` を指定: 404

## 9. 認証・認可設計

### 9.1 認証方式

Phase1 は server-side session table + Cookie session を採用する。

方針:

- login 成功時に session を作成
    
- session cookie は HttpOnly
    
- Secure / SameSite を設定する
    
- logout 時に DB session を失効する
    
- current user API で認証状態を確認する
    

### 9.2 session fixation 対策

login 成功時は session fixation 対策として、既存 session identifier を引き継がず、新しい session identifier を発行する。

logout 時は以下を行う。

- DB session を失効する
    
- session cookie を明示的に expire する
    
- CSRF token を無効化する
    

### 9.3 password 方針

- password は hash 化して保存する
    
- 平文 password は保存しない
    
- password hash は API response に含めない
    
- password hash は Argon2id を推奨する
    
- 採用が難しい場合は bcrypt を使用する
    
- hash parameter は実装規約書で固定する
    

### 9.4 login user enumeration 耐性

login 失敗時は、email 未存在 / password 不一致を区別しない。

方針:

- 同一の `error_code = auth_invalid_credentials`
    
- 同一の user safe message
    
- response timing の極端な差を避ける
    
- failed login audit を必要に応じて記録する
    

### 9.5 current user dependency

`get_current_user()` を用意する。

責務:

- session cookie 検証
    
- session 有効期限確認
    
- user 取得
    
- inactive user の拒否
    

### 9.6 role guard

`require_admin()` を用意する。

責務:

- current user の role を確認
    
- admin でなければ `PermissionDenied` を投げる
    

### 9.7 viewer scope

viewer は以下のみ許可される。

- 自身の chat session / messages
    
- 自身の user settings
    
- 通常 RAG ask
    
- temporary chat

## 10. CSRF 設計

### 10.1 基本方針

Cookie session を用いるため、state-changing API では CSRF token を必須とする。

対象:

- POST
    
- PUT
    
- PATCH
    
- DELETE
    

### 10.2 pre-auth CSRF

login 前 CSRF token は、pre-auth signed cookie と response body token を組み合わせて扱う。

Phase1 方針:

- `GET /api/v1/auth/csrf` は response body で `csrf_token` を返す
    
- frontend は response body の `csrf_token` を memory に保持する
    
- backend は pre-auth signed cookie を token 検証用 state として使う
    
- signed cookie は改ざん検知可能にする
    
- login 前の CSRF token は user session とは別に扱う
    

### 10.3 pre-auth CSRF cookie 属性

pre-auth CSRF 用 signed cookie は以下を基本とする。

- HttpOnly
    
- Secure
    
- SameSite=Lax
    
- short TTL
    
- signed
    
- tamper detection 必須
    

frontend が `X-CSRF-Token` に載せる token は response body から取得するため、pre-auth signed cookie 自体を JS から読む前提にはしない。

### 10.4 `/auth/csrf` response schema

`GET /api/v1/auth/csrf` は以下の形式で CSRF token を返す。

```json
{
  "data": {
    "csrf_token": "..."
  },
  "meta": {
    "request_id": "..."
  }
}
```

frontend は `data.csrf_token` を memory に保持し、state-changing API の `X-CSRF-Token` に設定する。

`GET /api/v1/auth/csrf` の response には以下の header を付与する。

```text
Cache-Control: no-store
```

CSRF token は browser / proxy / CDN に cache させない。

### 10.5 post-login CSRF

login 成功後は session identifier 再生成に合わせて、旧 CSRF token を無効化し、新 token を発行する。

login 成功時は pre-auth CSRF state / pre-auth signed cookie を失効し、session-bound CSRF state に切り替える。これにより、login 前 token が login 後 session で再利用されることを防ぐ。

### 10.6 CSRF token 発行

- CSRF token 取得 endpoint は `/api/v1/auth/csrf` とする
    
- login 前にも取得可能とする
    
- login 成功後は token を再発行する
    
- logout 後は token を無効化する
    

### 10.7 CSRF 検証

state-changing API では以下を検証する。

- header `X-CSRF-Token`
    
- session または pre-auth state に紐づく token
    
- token expiry
    

### 10.8 CSRF error

CSRF 不正時:

- `403`
    
- `error.code = csrf_invalid` または `csrf_missing`
    

### 10.9 retry 方針との整合

frontend 側では POST / PUT / PATCH / DELETE の CSRF error 後に自動 retry しない。backend も二重送信を許容しない前提で実装する。

## 11. API Router 設計

### 11.1 auth router

Prefix: `/api/v1/auth`

|method|path|auth|目的|
|---|---|---|---|
|GET|`/csrf`|public|csrf token 取得|
|POST|`/login`|public + csrf|login|
|POST|`/logout`|user + csrf|logout|

|method|path|auth|目的|
|---|---|---|---|
|GET|`/me/settings`|user|自身の設定取得|
|PATCH|`/me/settings`|user + csrf|自身の設定更新|

### 11.3 chat router

Prefix: `/api/v1/chat`

|method|path|auth|目的|

## 14. RAG `/rag/ask` 詳細

### 14.1 Phase1 実行方式

Phase1 の `/rag/ask` は同期実行を基本とする。

- API request 中に retrieval / rerank / generation / citation / confidence まで実行する
    
- message edit regeneration は job 経由で扱う
    
- `/rag/ask` の job 化は Phase2 以降の拡張候補とする
    

### 14.2 基本フロー

1. request validation
    
2. current user / session 権限確認
    
3. archived / temporary expired 判定
    
4. `client_message_id` duplicate 判定
    
5. user message 保存
    
6. retrieval_run 作成
    
7. retrieval
    
8. rerank
    
9. context 最小化 / PII masking
    
10. generation
    
11. citation 生成
    
12. confidence 算出
    
13. assistant message / citations / confidence 保存
    
14. `AskRagResultDTO` 組み立て
    
15. response 返却
    

### 14.3 retrieval_runs.status

`retrieval_runs.status` は Phase1 で正式に利用する。

Phase1 で使用する値:

- `running`
    
- `succeeded`
    
- `failed`
    

方針:

- `/rag/ask` / `/rag/search` 開始時に `running` として作成し、同時に `started_at = now()` を設定する
    
- 作成直後は `finished_at = NULL`、`error_code = NULL` とする
    
- 成功時に `succeeded` へ更新する
    
- 失敗時に `failed` へ更新する
    
- `queued` は Phase1 の同期実行では使用しない。将来 `/rag/ask` を job 化する場合の拡張候補とする
    

DDL v1.5 には、少なくとも以下の列が必要である。

- `retrieval_runs.status`
    
- `retrieval_runs.error_code`
    
- `retrieval_runs.started_at`
    
- `retrieval_runs.finished_at`
    

DDL v1.5 にこれらが存在しない場合は、DDL 側を追随修正する。

`retrieval_runs.status` の CHECK / enum は、本節の `running / succeeded / failed` と一致させる。

### 14.4 duplicate / replay 制御

キー:

```text
chat_session_id + client_message_id
```

#### 成功済み + 同一本文

- 既存結果を返す
    
- 新規 user message は作成しない
    
- 新規 retrieval_run は作成しない
    
- `meta.replayed = true`
    

#### 実行中

- `409 request_in_progress`
    

#### 失敗済み + 同一本文

- `409 conflict`
    
- 新しい `client_message_id` で再送を促す
    

#### 同じ client_message_id で異本文

- `409 conflict`
    

### 14.5 duplicate race condition 対策

`client_message_id` の重複制御は DB unique constraint を最終防衛線とする。

同時実行で `IntegrityError` が発生した場合:

1. rollback する
    
2. `chat_session_id + client_message_id` で既存 user message を再取得する
    
3. 本文一致 / 不一致を確認する
    
4. 関連 retrieval_run / assistant message の状態を確認する
    
5. 状態に応じて replay / request_in_progress / conflict に変換する
    

### 14.6 request_in_progress 判定条件

Phase1 の `/rag/ask` は同期実行を基本とするため、request_in_progress は `/rag/ask` の同一 `client_message_id` に対する処理中状態を表す。

以下を満たす場合、request_in_progress とみなす。

- user message は作成済み
    
- 対応する assistant message が未作成
    
- user message に紐づく retrieval_run が `running`
    

message edit regeneration など job 経由の処理は、edit lineage 側の in-progress 判定として扱う。

retrieval_run が `failed` かつ assistant message が存在しない場合は、失敗済みとして conflict 扱いにする。

### 14.7 failure boundary

#### retrieval 開始前失敗

retrieval_runs を作成しない。

例:

- validation error
    
- permission error
    
- archived session
    
- request_in_progress
    

#### retrieval 開始後失敗

retrieval_runs を残す。assistant message は作成しない。

例:

- vector DB failure
    
- rerank failure
    
- generation failure
    
- timeout
    

### 14.8 retrieval_run 失敗時更新保証

retrieval_run 作成後に失敗した場合は、error handling path で以下を更新する。

- `retrieval_runs.status = failed`
    
- `error_code`
    
- `finished_at`
    

この更新に失敗した場合も、application log に `request_id` 付きで記録する。

### 14.9 citation 保存整合

citations は、同一 `retrieval_run_id` の retrieval_run_items に含まれる `document_chunk_id` のみを参照して作成する。

これにより、実際に retrieval result に含まれていない chunk が citation として保存されることを防ぐ。

### 14.10 transaction 方針

RAG 処理は長時間化しうるため、外部 API 呼び出しを含む全体を 1 transaction にしない。

推奨保存単位:

1. user message / initial retrieval_run 保存
    
2. retrieval_run_items 保存
    
3. citations / assistant message / confidence / retrieval_runs.status 更新
    
4. 失敗時 retrieval_run status 更新
    

#### user message / initial retrieval_run

user message と initial retrieval_run は同一 short transaction で保存する。

この transaction が成功しない限り、user message 単独では commit しない。

理由:

- `request_in_progress` 判定を安定させる
    
- user message は存在するが retrieval_run が存在しない中間状態を避ける
    
- duplicate / replay 判定で必要な状態を揃える
    

#### 成功時最終 transaction

成功時は以下を同一 short transaction で保存する。

- citations
    
- assistant message
    
- `assistant_message.linked_retrieval_run_id = retrieval_run_id`
    
- confidence
    
- `retrieval_runs.status = succeeded`
    
- `retrieval_runs.finished_at`
    

これにより、assistant message は作成済みだが retrieval_run が running のまま残る不整合、および assistant message から採用 retrieval_run をたどれない trace 欠落を避ける。

#### 失敗時

retrieval_run 作成後に失敗した場合は、失敗 path で `retrieval_runs.status = failed` / `error_code` / `finished_at` を更新する。

retrieval_run_items 保存に失敗した場合も同様に retrieval_run を failed に更新し、assistant message は作成しない。

## 17. Job / Worker 詳細

### 17.1 jobs table polling

Phase1 は PostgreSQL jobs table + worker polling とする。

### 17.2 job lifecycle

- queued
    
- running
    
- succeeded
    
- failed
    
- canceled
    

### 17.3 lease acquisition

queued job 取得は `SELECT ... FOR UPDATE SKIP LOCKED` を用いる。

取得と以下の更新は同一 transaction で行う。

- `status = running`
    
- `locked_by`
    
- `locked_at`
    
- `lease_expires_at`
    
- `started_at`
    

これにより複数 worker による同一 job の二重取得を防ぐ。

### 17.4 reclaim

`lease_expires_at < now()` の running job を reclaim 対象とする。

方針:

- `FOR UPDATE SKIP LOCKED` で取得する
    
- 新 job は作らない
    
- retry lineage は増やさない
    
- 既存 job の `locked_by` / `locked_at` / `lease_expires_at` を更新する
    
- 必要に応じて reclaim 回数または application log を残す
    

### 17.5 retry

retry は failed job に対して新 job を作る。

- 元 job は failed のまま
    
- 新 job は queued
    
- `retry_of_job_id` を持つ
    
- active retry は 1 件まで
    

### 17.6 active retry 競合

active retry 競合は、アプリケーションチェックに加え、DB partial unique index を最終防衛線とする。

同時 retry で `IntegrityError` が発生した場合:

- rollback
    
- 既存 active retry を再取得
    
- `job_active_retry_exists` / 409 に変換
    

### 17.7 worker handler

|job_type|handler|
|---|---|
|document_ingest|ingest_handler|
|embedding|embedding_handler|
|evaluation|evaluation_handler|
|temporary_cleanup|cleanup_handler|
|message_edit_regenerate|message_edit_regenerate_handler|

## 18. Temporary Chat Cleanup 詳細

### 18.1 対象

temporary_cleanup は以下を対象とする。

```text
temporary_flag = true
AND ttl_expires_at < now()
```

### 18.2 transaction 単位

temporary cleanup は対象 `chat_session_id` 単位で transaction を張る。

方針:

- 対象 session を削除 transaction 内で行ロックし、同一 session に対する同時更新を避ける
    
- 1 session の削除中に失敗した場合は当該 session の削除を rollback する
    
- 次の対象 session へ処理を進める
    
- 失敗は `request_id` / `job_id` 付きで application log に記録する
    

### 18.3 linked_retrieval_run_id の扱い

temporary cleanup では、対象 chat_messages の `linked_retrieval_run_id` を NULL に更新してから retrieval 系テーブルを削除する。

方針:

1. 対象 session の chat_messages を特定する
    
2. 対象 chat_messages の `linked_retrieval_run_id` を NULL に更新する
    
3. citations / retrieval_run_items / retrieval_runs を削除する
    
4. chat_messages / chat_sessions を削除する
    

DDL の FK が `ON DELETE SET NULL` または `ON DELETE CASCADE` になっている場合でも、Phase1 アプリ実装では明示的に NULL 化してから削除する方針とする。

### 18.4 削除対象

TTL 到達後、以下を削除対象とする。

- chat_sessions
    
- chat_messages
    
- summary_memories
    
- chat_tags
    
- retrieval_runs
    
- retrieval_run_items
    
- citations
    

### 18.5 削除しないもの

以下は削除しない。

- audit_logs
    
- 運用・監査上必要な jobs
    

### 18.6 jobs の target 参照

temporary chat に関連する jobs を残す場合、削除済み target を参照する可能性がある。

方針:

- jobs は対象 chat_session への強い FK を持たない soft reference として扱う
    
- job payload / target_type / target_id に temporary chat の ID が残る可能性を許容する
    
- job detail 表示では、削除済み target を `target_deleted = true` 相当の表示情報へ変換する
    
- raw payload は UI に表示しない
    

### 18.7 evaluation との関係

temporary chat 起源の retrieval_run は evaluation 対象外とする。  
temporary chat 起源の evaluation 参照は作らない前提とする。

### 18.8 削除順序

FK 制約に従い、子テーブルから順に削除する。

基本順序:

1. 対象 chat_messages の `linked_retrieval_run_id` を NULL 化
    
2. citations
    
3. retrieval_run_items
    
4. retrieval_runs
    
5. summary_memories
    
6. chat_tags
    
7. chat_messages
    
8. chat_sessions

画像 OCR は Phase3 実装対象のため、`.png` / `.jpg` / `.jpeg` など画像系は Phase3 まで除外する。

### 19.3 新規 document upload

`POST /documents` は新規 logical document と初回 version を作成する。

### 19.4 既存 document version 追加

`POST /documents/{logical_document_id}/versions` は既存 logical document に新 version を追加する。

### 19.5 duplicate content skip

同一 `logical_document_id + content_hash` が存在する場合:

- version を増やさない
    
- ingest job は作成しない
    
- API response で success status `duplicate_content_skipped` を返す
    
- audit log に `action_type = document_duplicate_skipped` を残す
    
- success 系 response として返す
    

### 19.6 archive

logical document archive 時:

- `logical_documents.status = archived`
    
- active version を無効化
    
- retrieval 対象外
    

`logical_documents.status = archived` と `document_versions.is_active = false` は同一 transaction で更新する。

これにより、document は archived だが active version が残る、または active version だけ無効化された中間状態を避ける。

### 19.7 Qdrant payload mirror 更新

Phase1 では、document archive 時の Qdrant payload mirror 更新は job 作成方式とする。

方針:

- RDB の `logical_documents.status` / `document_versions.is_active` を正とする
    
- Qdrant payload の `is_active` は検索高速化用の denormalized mirror とする
    
- archive API は RDB 更新を先に確定する
    
- Qdrant payload mirror 更新 job を作成する
    
- retrieval 実行時は RDB の active / archived 状態を最終判定として扱う
    

これにより、Qdrant mirror 更新前でも archived document が retrieval 対象に混入しないようにする。

Qdrant mirror 更新 job が失敗しても、RDB が archived であれば retrieval 対象外判定は維持する。

失敗した Qdrant mirror 更新 job は jobs 画面で確認し、retry 可能とする。

### 19.8 PDF text extraction 失敗時

`.pdf` は Phase1 upload allowlist に含めるが、text extraction に失敗する可能性がある。

PDF text extraction に失敗した場合:

- `document_versions.status = failed`
    
- `error_code` を保存する
    
- ingest job は failed とする
    
- UI では取り込み失敗として表示する
    

`document_versions.status = failed` の version は approve 対象外とし、active version にはできない。復旧は再 upload または新 version 追加で行う。

### 19.9 physical delete

admin 限定。

条件:

- 参照済み文書は原則削除不可
    
- 参照済み文書を physical delete しようとした場合は 409 conflict
    
- 監査上の最小メタデータは残す

## 21. Audit / Logging 詳細

### 21.1 audit 対象

- login / logout
    
- document upload / approve / reject / archive / delete
    
- duplicate content skip
    
- evaluation run
    
- job retry
    
- system_settings update
    
- external API 利用のうち security relevant または管理上必要なイベント
    
- security relevant error
    

### 21.2 audit 必須項目

- request_id
    
- action_type
    
- target_type
    
- created_at
    

`target_id` は原則記録するが、login failure など対象リソースが存在しないイベントでは NULL を許容する、または synthetic target_id を使用する。DDL の nullable 方針に合わせる。

`target_id = NULL` の audit log も一覧・検索対象に含める。target_id filter が指定された場合は、NULL レコードは一致対象外とする。

### 21.3 actor_user_id

human actor が存在する API 起点イベントでは `actor_user_id` を保存する。

内部 job 起点イベントでは、DDL 方針に応じて以下のいずれかとする。

- `actor_user_id = NULL`
    
- system user を seed し、`actor_user_id = system_user_id`
    

Phase1 推奨は `actor_user_id = NULL` である。DDL が NOT NULL の場合は system user seed を採用する。

### 21.4 audit 条件付き項目

該当時のみ保存する。

- error_code
    
- job_id
    
- retrieval_run_id
    
- chat_session_id
    
- logical_document_id
    
- document_version_id
    

### 21.5 内部 job 起点

内部 job 起点では `system-generated request_id` を採番する。

### 21.6 external API 利用ログ

LLM / embedding / rerank などの外部 API 利用は、application log / metrics を基本とする。

audit log には、security relevant または管理上必要なイベントのみ記録する。

禁止事項:

- raw prompt を保存しない
    
- raw response を保存しない
    
- raw document text を保存しない
    
- API key / token を保存しない
    

### 21.7 redaction

audit log に raw PII / raw prompt / raw document text / raw payload_json を保存しない。

## 22. Security 詳細

### 22.1 PII 方針

- 原文保存は必要最小限
    
- 暗号化
    
- 厳格 ACL
    
- 外部 API 送信前に不可逆 mask
    
- 再識別が必要な項目のみ secure mapping store で reversible tokenization
    

### 22.2 raw payload 方針

以下は application log / audit log / UI に出さない。

- raw prompt
    
- raw document text
    
- raw payload_json
    
- API key
    
- token
    
- session secret
    
- password hash
    

### 22.3 file security

- MIME / magic bytes 検証
    
- 拡張子 allowlist
    
- size limit
    
- compression bomb 対策
    
- malware scan は Phase2 以降の候補
    

### 22.4 CORS

- allowed origins を settings で管理
    
- credentials を使うため wildcard は使わない
    

### 22.5 Cookie

- HttpOnly
    
- Secure
    
- SameSite
    
- session expiration
    

Phase1 の同一 site 前提では `SameSite=Lax` を基本とする。cross-site 構成が必要な場合のみ `SameSite=None + Secure` を検討する。

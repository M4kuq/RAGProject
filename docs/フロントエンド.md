## 2. 技術前提

### 2.1 Phase1 採用ライブラリ

Phase1 のフロントエンドは以下を正式採用する。

| 分類             | 採用                        | 用途                                              |
| -------------- | ------------------------- | ----------------------------------------------- |
| UI             | React                     | 画面実装                                            |
| 言語             | TypeScript                | 型安全性・保守性向上                                      |
| ルーティング         | React Router              | 画面遷移 / guard                                    |
| server state   | TanStack Query            | cache / loading / error / refetch / mutation 管理 |
| form           | React Hook Form           | form state / validation 管理                      |
| API 通信         | fetch wrapper + API hooks | API 呼び出し統一                                      |
| test runner    | Vitest                    | unit / hooks / component test                   |
| component test | React Testing Library     | UI 挙動テスト                                        |

### 2.2 採用理由

#### TanStack Query

本システムは一覧・詳細・mutation・refetch が多いため、TanStack Query に統一することで loading / error / cache / invalidation の実装を安定させる。

対象例:

* chat history
* chat messages
* document list / detail
* evaluation runs / detail
* jobs / detail
* audit logs
* system settings

#### React Hook Form

login、user settings、document upload、retrieval debug、audit filter などで form state と validation が必要になるため、Phase1 から採用する。

#### Vitest + React Testing Library

React / TypeScript 構成と相性がよく、component / hooks の挙動を軽量に検証できるため採用する。

## 4. Providers 設計

### 4.1 providers.tsx の責務

`app/providers.tsx` ではアプリ横断 provider を集約する。

配置候補:

* `QueryClientProvider`
* `AuthProvider`
* `ToastProvider`
* `ModalProvider`
* `RouterProvider`

### 4.2 provider 順序方針

基本順序は以下とする。

1. `QueryClientProvider`
2. `AuthProvider`
3. `ToastProvider`
4. `ModalProvider`
5. `RouterProvider`

理由:

* AuthProvider は `useCurrentUser()` など query を使う可能性がある
* Toast / Modal は route 内の UI から利用される
* Router は guard / page 表示を担う

### 4.3 Router hook 依存の制約

上記 provider 順序では `AuthProvider` が `RouterProvider` の外側にあるため、`AuthProvider` は `useNavigate()` などの Router hook に依存しない。

方針:

* AuthProvider は認証状態の取得・保持に責務を限定する
* login / logout 後の遷移は Page component、mutation 呼び出し元、または route guard 側で行う
* AuthProvider 内で route 遷移を直接実行しない

### 5.2 route 衝突対策

`/chat/:chatSessionId?` と `/chat/temp/:temporaryChatId?` は route 定義順によって衝突しうる。

Phase1 では以下の方針で実装する。

* `/chat/temp/:temporaryChatId?` を `/chat/:chatSessionId?` より優先して定義する
* React Router の nested route では `temp` segment を明示的に分離する
* `/chat/temp/...` が `chatSessionId = 'temp'` として解釈されないようにする

### 5.3 guard 方針

#### AuthGuard

* 未ログイン状態では `/login` に遷移する
* 認証状態確認中は full page loading を表示する

#### RoleGuard

* admin route は `role='admin'` のみ許可する
* viewer が admin route に直接アクセスした場合は `ForbiddenPage` を表示する
* viewer の navigation には admin route を表示しない
* 権限不足時に自動で通常チャットへ戻さない

### 5.4 auth status

AuthProvider はアプリ起動時に current user を取得し、以下の状態を持つ。

```ts
type AuthStatus = 'checking' | 'authenticated' | 'anonymous';
```

| auth status   | 表示                |
| ------------- | ----------------- |
| checking      | full page loading |
| anonymous     | `/login` redirect |
| authenticated | route 表示          |

これにより、初回ロード時に一瞬 login へ遷移する UI flicker を防ぐ。

### 5.5 403 / 404 表示基準

403 / 404 は以下の基準で表示を分ける。

| ケース                     | 表示                                       |
| ----------------------- | ---------------------------------------- |
| route 全体に対する 403        | `ForbiddenPage`                          |
| 画面内操作に対する 403           | inline error / toast                     |
| route 対象 resource の 404 | `NotFoundPage`                           |
| 画面内部分データの 404           | inline error / empty state / error state |

例:

* viewer が `/admin/documents` に直接アクセスした場合: `ForbiddenPage`
* 文書一覧内の archive 操作で 403 が返った場合: inline error / toast
* `/admin/documents/:logicalDocumentId` の対象が存在しない場合: `NotFoundPage`
* 文書詳細内の一部 chunk preview が存在しない場合: inline error / empty state

### 5.6 route mode 方針

`ChatPage` は route / fetched session state に応じて以下の mode を持つ。

| mode              | 条件                                        | UI制御                    |
| ----------------- | ----------------------------------------- | ----------------------- |
| active            | 通常 chat                                   | 入力可 / edit 可            |
| archived          | session.status = archived                 | 入力不可 / edit 不可 / 読み取り専用 |
| temporary         | temporary route または temporary_flag = true | TTL 有効中は入力可             |
| temporary_expired | TTL 期限切れ                                  | 入力不可 / edit 不可          |

## 7. 状態管理方針

### 7.1 状態分類

| 分類               | 例                                | 管理方法                      |
| ---------------- | -------------------------------- | ------------------------- |
| auth state       | current user / role / authStatus | AuthProvider + hooks      |
| server state     | chats / documents / jobs         | TanStack Query            |
| form state       | login / upload / settings        | React Hook Form           |
| UI state         | modal / toast / sidebar          | Context / local state     |
| optimistic state | sending message                  | chat feature local state  |
| csrf state       | csrf token                       | apiClient 内部または csrfStore |

### 7.2 server state

* API から取得する一覧・詳細は server state として扱う
* loading / error / refetch を hooks に集約する
* Page component は API 通信詳細を持たない
* mutation 後は query key 方針に従って cache update / invalidate を行う

### 7.3 client UI state

以下は client state とする。

* toast 表示
* confirm dialog
* sidebar open / close
* selected tab
* temporary optimistic message

### 7.4 optimistic message state

チャット送信時は、一時的に optimistic message を表示する。

必要項目:

* temporary id
* chat_session_id
* client_message_id
* body
* status: sending / persisted / failed_unknown

### 7.5 optimistic merge key

optimistic user message と persisted user message の突合キーは以下とする。

```text
chat_session_id + client_message_id
```

API 応答または messages 再取得時に同一 `client_message_id` の persisted message を検出した場合:

* optimistic temporary id を `chat_message_id` に置換する
* optimistic message と persisted message を重複表示しない
* body が異なる場合は API 契約上 conflict として扱う

## 8. API Client 設計

### 8.1 apiClient の責務

`lib/apiClient.ts` に共通 fetch wrapper を置く。

責務:

* base URL 付与
* credentials / Cookie 送信
* CSRF token 付与
* JSON encode / decode
* HTTP error の統一変換
* request_id の受け取り
* 401 / 403 / 404 / 409 / 422 / 503 のエラー型変換

### 8.2 error 型

共通エラー型を定義する。

```ts
type ApiError = {
  status: number;
  errorCode?: string;
  message: string;
  requestId?: string;
  details?: unknown;
};
```

### 8.3 request_id 取得方針

request_id は以下の優先順で取得する。

1. response header の `X-Request-Id`
2. response body の `meta.request_id`
3. どちらも存在しない場合は `undefined`

UI のエラー表示では、存在する場合のみ request_id を表示してよい。

### 8.4 HTTP status と UI 表示

| status | UI方針                              |
| ------ | --------------------------------- |
| 401    | ログインへ誘導                           |
| 403    | ForbiddenPage または権限がない旨を表示        |
| 404    | NotFoundPage または対象が存在しない旨を表示      |
| 409    | conflict / request_in_progress 表示 |
| 422    | validation error 表示               |
| 503    | 一時的障害として再試行案内                     |

## 9. CSRF 設計

### 9.1 基本方針

Cookie session + CSRF 前提のため、フロントエンドは CSRF token を扱う。

方針:

* 初回ロード時または login 前に CSRF token を取得する
* state-changing API では `X-CSRF-Token` を付与する
* login 成功後は response body の session-bound CSRF token に更新する
* logout 成功後は CSRF token を破棄する
* `csrf_missing` / `csrf_invalid` の場合は token を再取得する

対象となる state-changing API:

* POST
* PUT
* PATCH
* DELETE

### 9.2 CSRF token 保存場所

* CSRF token は localStorage に保存しない
* CSRF token は module-level memory、React state、または Context で保持する
* page reload 時は再取得する
* token は console / application log に出力しない

### 9.3 refresh 後の retry 方針

CSRF refresh 後の自動 retry は以下とする。

| API 種別                               | 自動 retry 方針                                                                                 |
| ------------------------------------ | ------------------------------------------------------------------------------------------- |
| GET 系                                | 通常 CSRF token を要求しない。ただし CSRF token 取得 API や認証状態確認 API で token 更新が必要な場合は、再取得後に自動 retry してよい |
| POST / PUT / PATCH / DELETE 系        | 原則自動 retry しない                                                                              |
| idempotent と明示された state-changing API | 個別に自動 retry を検討可                                                                            |

state-changing API で CSRF エラーになった場合:

1. token を再取得する
2. 自動再送はしない
3. ユーザーに再操作を促す

理由:

* `/rag/ask`
* document upload
* job retry
* archive / delete

など、副作用を持つ操作で二重送信を防ぐため。

### 9.4 CSRF refresh single-flight 方針

CSRF token refresh は single-flight とする。

方針:

* 同時に複数の refresh request を発行しない
* refresh 中に別 request が `csrf_invalid` を受けた場合、既存の refresh promise を待つ
* refresh 成功後、GET 系または許可された retry 対象のみ再実行する
* state-changing API は refresh 後も原則自動 retry しない

これにより、CSRF token の競合更新や refresh request の多重発行を防ぐ。

### 9.5 CSRF hooks

以下を用意する。

* `useCsrfToken()`
* `refreshCsrfToken()`

実装上は apiClient 内部で透過的に CSRF token を扱ってもよい。
ただし、責務としては `features/csrf/` または `lib/apiClient.ts` に閉じ込め、各 Page component に CSRF 実装を漏らさない。

## 10. TanStack Query 設計

### 10.1 QueryClient 方針

`lib/queryClient.ts` で QueryClient を初期化する。

基本方針:

* 一覧・詳細は query 管理
* 登録・更新・削除は mutation 管理
* mutation 成功後は cache update / invalidate を行う
* logout 時は query cache を clear

### 10.2 default retry 方針

TanStack Query の default retry は以下とする。

#### query

* `401 / 403 / 404 / 409 / 422` では retry しない
* `503` や一時的な network error は、回数を制限して retry を許容する
* retry 回数・間隔は実装規約書で固定する

#### mutation

* mutation は原則 retry しない
* `/rag/ask`、document upload、job retry、archive / delete など副作用を持つ操作は自動 retry しない
* 明示的に idempotent と判断できる mutation のみ、個別に retry を検討する

目的:

* validation error や permission error の無意味な再実行を防ぐ
* duplicate / conflict の自動再送を防ぐ
* state-changing API の二重実行を防ぐ

### 10.3 query keys

`lib/queryKeys.ts` に query key factory を定義する。

| key                                                | 用途                  |
| -------------------------------------------------- | ------------------- |
| `['currentUser']`                                  | current user        |
| `['chatSession', chatSessionId]`                   | chat session detail |
| `['chatMessages', chatSessionId, page, page_size]` | chat messages       |
| `['chatHistory', normalizedParams]`                | chat history        |
| `['documents', normalizedParams]`                  | document list       |
| `['documentDetail', logicalDocumentId]`            | document detail     |
| `['evaluationRuns', normalizedParams]`             | evaluation run list |
| `['evaluationRun', evaluationRunId]`               | evaluation detail   |
| `['jobs', normalizedParams]`                       | job list            |
| `['job', jobId]`                                   | job detail          |
| `['auditLogs', normalizedParams]`                  | audit logs          |
| `['systemSettings']`                               | system settings     |

### 10.4 query key params 正規化

params object を query key に含める場合は、hooks 内または query key factory で正規化する。

方針:

* default 値を埋める
* `undefined` と省略の揺れをなくす
* page / page_size を明示する
* filter 未指定は `'all'` や `null` などに統一する

例:

```ts
queryKeys.documents({
  page,
  page_size,
  status: status ?? 'all',
});
```

### 10.5 mutation invalidation / cache update 方針

| mutation                       | cache update / invalidate                                                                                                             |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------- |
| `useAskRag` 成功                 | 現在表示中の最新ページの chatMessages cache に user / assistant を反映。chatSession / chatHistory は invalidate。chatMessages は必要に応じて background refetch |
| `useEditMessage` 成功            | user message 本文を即時反映。jobs を invalidate。chatMessages は job terminal 後に refetch                                                         |
| `useArchiveChat` 成功            | 対象 detail / list の status を optimistic に archived へ更新してよい。その後 chatHistory, chatSession を invalidate                                   |
| `useCreateTemporaryChat` 成功    | temporary session / messages の query を初期化し、UI local state に保持。chatHistory は invalidate しない                                            |
| `useUploadDocument` 成功         | documents, jobs を invalidate                                                                                                          |
| `useApproveDocumentVersion` 成功 | documents, documentDetail, jobs を invalidate                                                                                          |
| `useRejectDocumentVersion` 成功  | documents, documentDetail を invalidate                                                                                                |
| `useArchiveDocument` 成功        | 対象 detail / list の status を optimistic に archived へ更新してよい。その後 documents, documentDetail を invalidate                                  |
| `useDeleteDocument` 成功         | documents を invalidate                                                                                                                |
| `useCreateEvaluationRun` 成功    | evaluationRuns, jobs を invalidate                                                                                                     |
| `useRetryJob` 成功               | jobs, job を invalidate                                                                                                                |
| `useUpdateSystemSetting` 成功    | systemSettings を invalidate                                                                                                           |

### 10.6 `useAskRag` の特別方針

`useAskRag` は optimistic UI と競合しやすいため、単純な immediate invalidate のみにしない。

成功時:

1. optimistic user message を persisted message に置換する
2. assistant message を、現在表示中の最新ページの `chatMessages` query に反映する
3. citations / confidence を cache に反映する
4. chatSession / chatHistory は invalidate する
5. chatMessages は必要に応じて background refetch とする

古いページ閲覧中に送信した場合は、送信前に最新ページへ移動してから mutation を実行する。

目的:

* UI のちらつきを抑える
* optimistic message と persisted message の重複を防ぐ
* pagination meta は background refetch で整合させる

### 10.7 archive no-op success の扱い

archive 成功時は対象 detail / list の status を optimistic に `archived` へ更新してよい。

no-op success の場合も、UI ではエラー扱いせず、以下のように表示する。

* chat archive: 「既にアーカイブ済み」
* document archive: 「既にアーカイブ済み」

最終整合は invalidate / refetch により取る。

### 10.8 `useEditMessage` の特別方針

edit は「job 作成成功」と「lineage 再生成成功」を分けて扱う。

#### useEditMessage 成功時

* edit job 作成成功として扱う
* user message 本文を即時反映する
* jobs を invalidate する
* 再生成中 banner を表示する

#### job succeeded 確認後

* chatMessages を refetch する
* 新 lineage に切り替える
* 再生成中 banner を消す

#### job failed 確認後

* 旧 lineage を維持する
* 再生成失敗を表示する

### 10.9 logout 時 cache clear

logout 成功時は以下を行う。

* TanStack Query の cache を clear する
* auth state を anonymous に戻す
* CSRF token を破棄する
* `/login` へ遷移する

Cookie session のため、logout 後に chat history / messages / documents / audit logs などの cached server state が残らないようにする。

### 10.10 logout API 失敗時

logout API が失敗した場合でも、セキュリティを優先してローカル状態を破棄する。

実施すること:

* auth state を anonymous に戻す
* TanStack Query cache を clear する
* CSRF token を破棄する
* `/login` へ遷移する
* 必要に応じて「サーバー側ログアウト状態を確認できませんでした」と表示する

## 11. Pagination / chat messages 設計

### 11.1 pagination 型

API の naming に合わせ、フロント側も `page_size` を使用する。

```ts
type PaginationParams = {
  page: number;
  page_size: number;
};

type PaginationMeta = {
  page: number;
  page_size: number;
  total: number;
  has_next: boolean;
};
```

`PaginationMeta` のフィールド名は API設計書 v1.6 の `meta.pagination` と一致させる。

Phase1 では API 側の `total` / `has_next` に合わせる。

### 11.2 chat message pagination 方針

ChatPage は原則「最新ページ」を表示する。

方針:

* optimistic message は現在表示中の最新ページ末尾に追加する
* 古いページ閲覧中に送信操作はしない、または送信時に最新ページへ移動する
* persisted sync 後、pagination の total は background refetch で整合させる
* older messages の閲覧は pagination により行う

### 11.3 古いページ閲覧中の送信

Phase1 では、古いページ閲覧中に送信操作した場合は最新ページへ移動してから送信する。

理由:

* optimistic message の挿入位置を単純化する
* pagination meta との不整合を避ける
* テストしやすくする

### 11.4 temporary chat cache 方針

TTL 期限切れを確認した temporary chat は、該当 temporary session / messages の query cache を remove または無効化する。

方針:

* temporary_expired 確認後は、以後の UI を期限切れ表示へ寄せる
* temporary chat は恒久履歴一覧に含めないため、chatHistory は invalidate しない
* 必要に応じて local UI state からも temporary session を除去する

## 12. 共通型設計

### 12.1 API type と ViewModel の分離

API response をそのまま UI component に渡さず、必要に応じて ViewModel に変換する。

例:

```ts
type ChatMessageApi = {
  chat_message_id: string;
  chat_session_id: string;
  role: 'user' | 'assistant' | 'system';
  body: string;
  linked_retrieval_run_id?: string;
  created_at: string;
};

type ChatMessageViewModel = {
  id: string;
  role: 'user' | 'assistant' | 'system';
  body: string;
  createdAt: string;
};
```

内部 trace 表示可否は ViewModel に boolean として持たせず、mapper の `ViewContext` から派生させる。

```ts
type ViewContext = {
  role: 'viewer' | 'admin';
  surface: 'chat' | 'debug';
};
```

方針:

* viewer 用 ViewModel には `linked_retrieval_run_id` や `retrieval_run_id` を含めない
* 通常 chat surface では admin でも内部 trace ID を常時表示しない
* debug surface のみ trace ID 表示を許可する

### 12.2 mapper 方針

`features/chat/chatMappers.ts` に以下のような mapper を置く。

* `toChatMessageViewModel()`
* `toCitationViewModel()`
* `toJobViewModel()`

これにより、security display 方針を型・変換層でも支える。

## 14. Form validation 方針

### 14.1 基本方針

バックエンド validation を正としつつ、UX 向上のためフロントでも最低限の validation を行う。

### 14.2 validation 対象

| 項目                   | フロント validation                   |
| -------------------- | --------------------------------- |
| email                | required / email format           |
| password             | required                          |
| message text         | required / max length は API 設計に従う |
| memory_message_limit | 1〜20                              |
| ui_theme             | light / dark / system             |
| top_k                | 1〜20                              |
| audit date range     | 31日以内                             |
| upload file          | required / file size / extension  |

### 14.3 upload file validation の責務分担

frontend は UX 目的で以下を検証する。

* required
* file size
* extension

backend を正とする検証:

* magic bytes
* file signature
* MIME 厳密検証
* 危険ファイル判定
* 圧縮爆弾対策
* 実ファイル内容に基づく validation

frontend validation を通過しても、backend validation に失敗する可能性がある。UI は backend からの validation error を表示できるようにする。

### 14.4 validation error 表示

* field error は入力項目の近くに表示する
* form 全体のエラーは inline error または toast に表示する
* 422 response は field error に変換可能な場合は field に紐づける

## 15. チャット実装詳細

### 15.1 client_message_id 生成方式

`client_message_id` はフロントエンドで生成する。

方針:

* `crypto.randomUUID()` を使用する
* UUID に含まれる `-` は API validation 上許容する
* Phase1 対象ブラウザでは `crypto.randomUUID()` を必須前提とし、fallback は設けない
* 同一送信に対して同じ `client_message_id` を再利用しない
* failed ask の再送時も新しい `client_message_id` を生成する

### 15.2 message rendering

`MessageList` は message array を受け取り、role ごとに表示 component を切り替える。

* user: UserMessage
* assistant: AssistantMessage
* system: SystemMessage 相当。Phase1 では表示しないか、debug 表示のみ

### 15.3 assistant message 表示

assistant message は以下を表示する。

* answer body
* confidence label
* citation panel toggle
* 旧版ソース badge

### 15.4 citation panel

表示項目:

* display_label
* snippet
* page
* source_type
* 旧版 badge

viewer には内部 ID を表示しない。

### 15.5 confidence 表示

`confidence_label` を表示する。

表示例:

* High
* Medium
* Low

Phase1 では過度な装飾をせず、回答信頼度を理解しやすくする程度に留める。

### 15.6 旧版ソース表示

回答 citations の少なくとも1件が active version ではない document_version を指す場合、通常 UI でも「旧版ソースを含む」表示を行う。

表示位置は以下のいずれかとする。

* assistant message 上部
* citation panel 上部
* badge

## 16. `/rag/ask` optimistic UI 詳細

### 16.1 送信直後

1. `client_message_id` を生成
2. optimistic user message を追加
3. assistant loading row を追加
4. `useAskRag()` を実行

### 16.2 成功時

* optimistic user message を persisted message に同期
* assistant loading row を assistant message に置換
* citations / confidence を表示
* `meta.replayed = true` でも通常回答として扱う

### 16.3 失敗時

* user message は残す
* assistant loading row は削除する
* inline error または toast を表示する
* 再読み込み後は user message のみ表示する

### 16.4 保存可否不明時

通信断などで保存可否が不明な場合:

* message list を再取得する
* 保存済み message があれば optimistic message と統合する
* 保存が確認できない場合は再送導線を表示する

### 16.5 request_in_progress

* 新規 message を追加しない
* 「現在処理中」と表示する
* 同じ送信を増やさない

## 24. テスト観点

### 24.1 チャット画面

* optimistic user message が表示される
* 成功時に assistant message が表示される
* optimistic message が persisted message と重複しない
* failed ask でも user message が残る
* failed ask 再送時に新しい client_message_id が生成される
* replay 成功時に通常回答表示になる
* `meta.replayed = true` でも特別成功表示が出ない
* request_in_progress で新規 message が追加されない
* archived mode で入力不可になる
* temporary expired で入力不可になる
* 古いページ閲覧中に送信すると最新ページへ移動する

### 24.2 権限制御

* viewer に AdminSidebar が表示されない
* viewer が `/admin/documents` にアクセスすると ForbiddenPage が表示される
* admin は debug view にアクセスできる

### 24.3 認証 / logout

* 起動時 checking では full page loading が表示される
* anonymous では login に redirect される
* logout 後に query cache が clear される
* logout API が失敗しても local auth state / cache / CSRF token が破棄される
* logout 後に admin data が表示されない

### 24.4 管理画面

* raw payload_json が表示されない
* PII が表示されない
* failed job のみ retry 可能
* active retry がある場合 retry button が disabled になる
* audit logs で 31日超の date range が validation error になる

### 24.5 CSRF

* state-changing API に CSRF token が付与される
* csrf_invalid 時に token refresh が行われる
* CSRF token refresh が single-flight である
* POST / PUT / PATCH / DELETE では csrf_invalid 後に自動 retry されない
* login 後に CSRF token が再取得される
* CSRF token が localStorage に保存されない

### 24.6 query / cache

* useAskRag 成功時に cache が重複表示を起こさない
* useAskRag 成功時の cache 反映対象が最新ページである
* useEditMessage 成功時は job 作成成功として扱われる
* job succeeded 後に chatMessages が refetch される
* params object の query key が正規化される
* archive no-op success がエラー扱いされない
* TanStack Query の default retry で 401 / 403 / 404 / 409 / 422 が retry されない
* mutation が原則 retry されない

### 24.7 routing

* route 一覧・route 定義の両方で `/chat/temp/:temporaryChatId?` が `/chat/:chatSessionId?` より優先される
* `/chat/temp/:temporaryChatId?` が `/chat/:chatSessionId?` に誤って一致しない
* viewer が admin route に直接アクセスすると ForbiddenPage が表示される

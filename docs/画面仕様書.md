## 1. 文書概要

### 1.1 目的

本書は、RAGシステム Phase1 の Web UI / CLI UI 実装に向けて、画面単位の目的・表示項目・操作・状態表示・権限差を定義する。

本書は詳細デザイン仕様ではなく、React 実装前に迷いやすい画面挙動を固定するための軽量画面仕様書である。

### 1.2 前提文書

本書は以下を前提とする。

- 要件定義書 v1.1
    
- 基本設計書 v1.2
    
- API設計書 v1.6
    
- ER図 / テーブル設計書 v1.4
    
- 状態遷移仕様書 v1.1
    
- DDL草案 v1.5
    
- 機能仕様補完書 v1.7
    

### 1.3 本書で決めること

- 画面一覧
    
- 画面ごとの目的
    
- 主な表示項目
    
- 主な入力項目
    
- 利用 API
    
- loading / empty / failed / archived の表示
    
- admin / viewer の画面差
    
- debug view の位置づけ
    

### 1.4 本書で決めないこと

- 具体的な CSS
    
- 余白 / 色 / アイコンの詳細
    
- コンポーネント実装詳細
    
- OpenAPI の完全な request / response schema

## 2. 画面全体方針

### 2.1 UI 方針

- Phase1 は機能説明性と実装容易性を優先する
    
- ChatGPT 風の会話 UI を基本とする
    
- admin 画面と通常チャット画面は明確に分離する
    
- debug view は admin 限定とし、通常チャット導線には出さない
    
- viewer には内部 trace ID を表示しない
    

### 2.2 状態表示方針

各画面は最低限以下の状態を持つ。

- loading
    
- empty
    
- error
    
- archived
    
- processing
    
- success / no-op success
    

### 2.3 権限差方針

#### viewer

- 通常チャット利用
    
- 通常会話履歴利用
    
- temporary chat 利用
    
- 自分の archived session 参照
    
- 自身の user settings 更新
    

#### admin

viewer 権限に加えて以下を利用できる。

- 文書管理
    
- evaluation 実行 / 閲覧
    
- job 状態閲覧
    
- audit log 閲覧
    
- system_settings 更新
    
- standalone retrieval debug view
    

### 2.4 画面モード方針

- UI-02 チャット画面は、active chat / archived mode / temporary mode の表示基盤を兼ねる
    
- UI-04 archived chat 詳細は、UI-02 の読み取り専用 mode として扱う
    
- UI-05 temporary chat 画面は、UI-02 の temporary mode として扱う
    
- 画面仕様上は説明のため分けるが、React 実装では共通チャットコンポーネントの mode 差分として実装してよい

# 5. UI-02 チャット画面

## 5.1 目的

ユーザーが質問を入力し、RAG回答・citation・confidence を確認する。

## 5.2 表示項目

- 会話タイトル
    
- message list
    
- user message
    
- assistant message
    
- citation panel
    
- confidence label
    
- 入力欄
    
- 送信ボタン
    
- 編集ボタン
    
- archive ボタン
    
- temporary chat 作成導線
    

## 5.3 入力項目

- message text
    
- client_message_id はフロントで生成する
    

## 5.4 client_message_id 生成ルール

- 新規送信ごとに新しい `client_message_id` を生成する
    
- failed ask を再送する場合も新しい `client_message_id` を生成する
    
- `request_in_progress` の場合は同じ送信を増やさない
    
- 同一 `client_message_id` で異本文を送らない
    

## 5.5 送信中 / optimistic 表示

通常送信直後は以下の表示とする。

1. user message を即時表示する
    
2. assistant 応答欄に loading 表示を出す
    
3. 成功時は assistant message を追加または loading 行を置換する
    
4. 失敗時は user message を残し、assistant loading を消す
    
5. 失敗時は inline error または toast を表示する
    

この挙動は、機能仕様補完書の「failed ask でも user message は残す」方針に合わせる。

## 5.6 optimistic 表示後のサーバー同期

- API 応答後は、サーバーから返された `message_id` / persisted message に UI 状態を同期する
    
- optimistic 表示中の一時 ID は、保存済み message の ID に置き換える
    
- 通信断などで保存可否が不明な場合は、再取得または再送導線を表示する
    
- 再取得により保存済み user message が確認できた場合は、重複表示しない
    

## 5.7 `/rag/ask` 成功時表示

- user message を表示
    
- assistant message を表示
    
- citation panel を表示可能にする
    
- confidence label を表示する
    
- citation に旧版が含まれる場合は「旧版ソースを含む」表示を行う
    

## 5.8 `/rag/ask` 失敗時表示

### 初回送信直後

- user message は表示する
    
- assistant message は表示しない
    
- inline error または toast を表示する
    

### 再読み込み後

- user message のみ表示する
    
- 失敗バナーや toast は保持しない
    

## 5.9 replay 表示

- `meta.replayed = true` の場合も通常回答として表示する
    
- 「再送成功」「再生成完了」のような特別成功表示は出さない
    
- 開発者向け debug 表示では replay であることを表示してよい
    

## 5.10 request_in_progress 表示

- `409 request_in_progress` の場合は「現在処理中」と表示する
    
- 新規 message は追加しない
    
- 送信ボタンは一定時間 disabled または再押下抑止する
    

## 5.11 duplicate / conflict 表示

- 同一 `client_message_id` で異本文または失敗済み再送の場合は conflict として表示する
    
- ユーザーには「再送してください」または「画面を更新して再試行してください」と案内する
    

## 5.12 message edit 表示

### edit 開始時

- 対象 user message 本文を即時反映してよい
    
- edit job 完了前は旧 downstream messages が暫定表示されうる
    
- UI は再生成中バナー / loading state / updating badge のいずれかを表示する
    

### edit 成功時

- 新 lineage を通常表示へ切り替える
    

### edit 失敗時

- 更新済み user message は残す
    
- 旧 lineage は壊さない
    
- 再生成失敗メッセージを表示する
    

## 5.13 archived mode 表示

- archived session は参照可能
    
- archived session では入力欄を disabled にする
    
- 送信ボタンを disabled にする
    
- edit ボタンを disabled または非表示にする
    
- 「この会話はアーカイブ済みです」と表示する
    

## 5.14 temporary mode 表示

- temporary badge を表示する
    
- 恒久履歴一覧には含めない
    
- 同一 UI セッション中は一時的に表示してよい
    
- TTL 有効中は通常送信可能
    
- TTL 期限切れ後は入力欄と送信ボタンを disabled にする
    
- TTL 期限切れ後の再送 / edit は不可
    
- TTL 期限切れ後にアクセスされた場合は「この一時チャットは期限切れです」と表示する
    

## 5.15 viewer / admin 差

### viewer

以下は表示しない。

- `retrieval_run_id`
    
- `linked_retrieval_run_id`
    
- `job_id`
    
- internal trace token
    

### admin

- 通常チャット画面では内部 ID を常時表示しない
    
- 必要な場合は debug view へ遷移して確認する

# 6. UI-03 会話履歴一覧

## 6.1 目的

過去の通常会話を一覧表示し、会話を再開する。

## 6.2 表示項目

- 会話タイトル
    
- 最終更新日時
    
- pinned 状態
    
- archived 状態
    
- タグ
    
- temporary でない通常会話のみ
    

## 6.3 表示対象

- active chat
    
- archived chat は通常一覧から分離してもよい
    
- temporary chat は恒久履歴一覧に含めない
    

## 6.4 ページング

Phase1 では pagination に固定する。

理由:

- API 整合が取りやすい
    
- テストしやすい
    
- infinite scroll より状態管理が単純
    

## 6.5 検索 / フィルタ

Phase1 では以下を候補とする。

- title search
    
- tag filter
    

## 6.6 empty 表示

会話がない場合は「まだ会話がありません」と表示する。

## 6.7 archived 表示

archived chat は、通常一覧とは別タブまたはフィルターで参照可能にする。  
archived chat を開く場合は UI-02 の archived mode として表示する。

## 6.8 no-op 表示

既に archived の chat を archive した場合、エラーではなく「既にアーカイブ済み」と表示する。

# 8. UI-05 temporary chat 画面

## 8.1 位置づけ

UI-05 は独立した実装画面ではなく、UI-02 チャット画面の temporary mode として扱う。

## 8.2 目的

履歴に残さない一時チャットを行う。

## 8.3 表示項目

- temporary badge
    
- message list
    
- input
    
- TTL / 残り時間表示は Phase1 では任意
    

## 8.4 一覧挙動

- temporary chat は恒久履歴一覧に表示しない
    
- 同一 UI セッション中は一時的に表示してよい
    

## 8.5 TTL 有効中

- 通常送信可能
    
- message edit も通常 chat と同じ制約で可能
    

## 8.6 TTL 到達後

- 会話・検索トレース系は削除対象
    
- audit_logs は削除対象に含めない
    
- jobs は運用・監査上必要な範囲で残してよい
    
- 入力欄と送信ボタンを disabled にする
    
- 再送 / edit は不可
    

## 8.7 UI 表示

TTL 期限切れ後にアクセスされた場合は「この一時チャットは期限切れです」と表示する。

- upload

## 16.7 citation 扱い

## 16.8 source label

- 複数版が混在しうる場合は版識別可能にする

# 19. 画面横断ルール

# 20. 総括

本書では、Phase1 の UI 実装で迷いやすい画面単位の挙動を軽量に定義した。

# AI/LLMエンジニア向けポートフォリオ提出用 RAGシステム

## 状態遷移仕様書 v1.1

---

## 1. 文書概要

### 1.1 目的

本書は、要件定義書 v1.1、基本設計書 v1.1、ER図 / テーブル設計書 v1.4、API設計書 v1.6 をもとに、Phase1 実装に必要な状態遷移を定義する。

本書の目的は以下である。

* 状態を持つ主語ごとの状態一覧と遷移ルールを固定する
* API / job / internal event のどれが状態を変化させるかを明確にする
* 遷移条件、遷移不可条件、副作用、監査要否を明記する
* DDL草案および実装時の再解釈を防ぐ

### 1.2 対象範囲

Phase1 で優先して状態遷移を定義する対象は以下とする。

* `chat_sessions`
* `chat_messages` と edit lineage
* `logical_documents`
* `document_versions`
* `jobs`
* `evaluation_runs`
* `evaluation_run_items`

加えて、以下の横断フローを対象とする。

* temporary chat TTL 到達時の削除フロー
* archive 実行時の retrieval 対象外化
* `/rag/ask` の success / in-progress / failed replay ルール
* `/rag/search` の standalone retrieval run 生成ルール

### 1.3 非対象

本書では以下は対象外とする。

* OpenAPI の詳細 schema
* SQLAlchemy 実装コード
* UI コンポーネント仕様詳細
* worker の内部アルゴリズム詳細

## 2. 状態遷移設計の共通原則

### 2.1 原則

* 状態は **明示的な trigger** によってのみ変化する
* trigger は `API` / `job` / `internal event` のいずれかに分類する
* 遷移不可時は API レベルでは `409 conflict` または `404`、job レベルでは `failed` とする
* Phase1 は DB trigger に過度に依存せず、**アプリケーション制御を正** とする
* 状態遷移時の副作用は、関連テーブル更新・監査ログ・Qdrant 更新まで含めて定義する

### 2.2 用語

* **遷移トリガー**: 状態を変化させる操作
* **副作用**: 状態変更に伴い発生する関連更新
* **横断ルール**: 単一テーブルでは閉じない運用ルール

### 2.3 監査対象の考え方

以下は原則として監査対象とする。

* 認証系状態変化
* archive / delete / approve / reindex
* evaluation 実行
* system settings 変更
* external API 利用

## 3. chat_sessions 状態遷移

## 3.1 状態一覧

* `active`
* `archived`

### 初期状態

* `active`

### 終端状態

* `archived`
* temporary chat の場合は TTL 到達後に物理削除されるため、DB上の終端状態は持たない

## 3.2 遷移表

| 現在状態     | 次状態      | トリガー                               | 種別  | 条件                           |
| -------- | -------- | ---------------------------------- | --- | ---------------------------- |
| なし       | active   | `POST /chat/sessions`              | API | 認証済み                         |
| active   | archived | `POST /chat/sessions/{id}/archive` | API | owner であること、temporary ではないこと |
| archived | archived | `POST /chat/sessions/{id}/archive` | API | no-op 許容                     |

## 3.3 遷移条件

* `temporary_flag = true` の session は通常 archive API の対象外とする
* archived session は参照可能だが、edit 系操作は不可とする

## 3.4 副作用

### 作成時

* owner_user_id を設定
* title 未指定時はサーバーが仮タイトルを補完
* temporary の場合は `ttl_expires_at` を設定

### archive 時

* session 自体の `status = archived`
* `updated_at` 更新
* 監査ログ追加

## 3.5 遷移不可時の扱い

* owner 不一致: `404`
* temporary session を archive しようとした場合: `409 conflict`

## 3.6 補足

* Phase1 では `deleted` 状態は持たない
* temporary chat は status 遷移で表現せず、TTL 到達時の物理削除で扱う

## 4. chat_messages / edit lineage 状態遷移

## 4.1 chat_messages 自体の状態

`chat_messages` は status 列を持たないため、**lineage と生成結果** で振る舞いを管理する。

### 実質的な分類

* 通常 message
* 編集済み message (`edited_flag = true`)
* 再生成により旧 lineage 側へ残った message
* 新 lineage に属する message

## 4.2 初期生成

| トリガー            | 種別       | 結果                   |
| --------------- | -------- | -------------------- |
| `/rag/ask` 正常受付 | API      | user message 作成      |
| 回答生成成功          | internal | assistant message 作成 |
| `/rag/search`   | API      | chat_message は作成しない  |

## 4.3 edit lineage 遷移

### トリガー

* `POST /chat/messages/{chat_message_id}/edit`

### 条件

* 対象は `role='user'`
* owner の session に属すること
* session が `active`
* 同一 message に対する edit job が `queued/running` 中でないこと

### 遷移内容

1. 対象 user message の `body` を更新
2. 対象 user message の `edited_flag = true`
3. edit job を作成
4. edit job 完了時、その message 以降を **新 lineage** として再生成
5. 新 lineage 側の assistant message 生成に失敗した場合でも、旧 lineage は変更しない

## 4.4 遷移表（概念）

| 現在              | 次                           | トリガー                            | 種別           |
| --------------- | --------------------------- | ------------------------------- | ------------ |
| 通常 user message | 編集済み user message           | `POST /chat/messages/{id}/edit` | API          |
| 旧 lineage       | 旧 lineage のまま保持             | edit job 完了                     | internal     |
| なし              | 新 lineage assistant message | edit job 完了後の再生成成功              | job/internal |

## 4.5 `/rag/ask` 失敗時の扱い

* user message は保存する
* assistant message は保存しない
* failed assistant placeholder は作らない
* user message は通常 `/messages` に表示する
* UI は request_id や API 応答に基づいて失敗表示してよい
* 再送時は **新しい `client_message_id` を持つ新しい user message** を追加する
* 旧 failed request と新 request は Phase1 では自動結合しない
* 入力不正や業務条件不成立は `422`、外部依存や一時的処理不能は `503` とする

## 4.6 `/rag/ask` replay / duplicate ルール

### 判定キー

* `chat_session_id + client_message_id`

### 分岐

* 成功済み + 同一本文: 既存結果返却
* 実行中: `409 request_in_progress`
* 失敗済み + 同一本文: `409 conflict`、新 `client_message_id` で再送
* 同一 `client_message_id` だが異本文: `409 conflict`

### 副作用

* duplicate request では新しい user message を作らない

## 4.7 補足

* `client_message_id` は user message のみ対象
* 旧 lineage は `include_internal_lineage=true` の場合のみ取得対象とする

## 5. logical_documents 状態遷移

## 5.1 状態一覧

* `active`
* `archived`

### 初期状態

* `active`

## 5.2 遷移表

| 現在状態     | 次状態      | トリガー                           | 種別  | 条件       |
| -------- | -------- | ------------------------------ | --- | -------- |
| なし       | active   | `POST /documents`              | API | admin    |
| active   | archived | `POST /documents/{id}/archive` | API | admin    |
| archived | archived | `POST /documents/{id}/archive` | API | no-op 許容 |

## 5.3 archive 時の副作用

* logical document を **retrieval 対象外** とする
* Phase1 では関連する **active version を無効化** する
* Qdrant payload の `is_active` mirror を更新する
* 監査ログを記録する

## 5.4 遷移不可時の扱い

* admin 以外: `403`
* 対象なし: `404`

## 5.5 補足

* Phase1 の通常 API フローでは archive は **logical document 単位** で扱う
* `document_versions.status='archived'` は Phase1 の通常運用では原則使用しない

## 6. document_versions 状態遷移

## 6.1 状態一覧

* `uploaded`
* `queued`
* `parsing`
* `parsed`
* `chunking`
* `embedding`
* `indexing`
* `ready`
* `failed`
* `pending_review`（表示・論理状態。DB status としては保持しない）
* `archived`

### 初期状態

* 新規アップロード直後は `uploaded`

## 6.2 標準 ingest 遷移

| 現在状態                                | 次状態       | トリガー         | 種別           |
| ----------------------------------- | --------- | ------------ | ------------ |
| uploaded                            | queued    | upload 受付完了  | API/internal |
| queued                              | parsing   | worker 開始    | job          |
| parsing                             | parsed    | parse 完了     | job          |
| parsed                              | chunking  | chunk 処理開始   | job          |
| chunking                            | embedding | chunk 完了     | job          |
| embedding                           | indexing  | embedding 完了 | job          |
| indexing                            | ready     | index 完了     | job          |
| parsing/chunking/embedding/indexing | failed    | 処理失敗         | job          |

## 6.3 review 経由遷移

folder watch / review 要件を考慮した場合:

| 現在状態           | 次状態            | トリガー             | 種別                          |
| -------------- | -------------- | ---------------- | --------------------------- |
| `processing`   | `ready + is_active=false` | 取り込み完了後 review 待ち | worker |
| `ready + is_active=false` | `ready + is_active=true` | approve API | API |
| `ready + is_active=false` | `failed` | reject / invalid | internal or admin operation |

## 6.4 active version ルール

* `is_active = true` にできるのは `status = 'ready'` の版のみ
* 1 logical_document あたり active version は 1件のみ
* 新しい version が active になる際は、既存 active version を false に落とす

## 6.5 archive との関係

* Phase1 の通常 API フローでは logical document 単位の archive を正とする
* logical document archive 時は、関連する active version を false にする
* `document_versions.status='archived'` は Phase1 では原則使わない

## 6.6 Phase1 における `archived` の位置づけ
- `document_versions.status='archived'` は将来拡張余地として状態一覧には残す
- Phase1 の通常運用では遷移対象外とする
- Phase1 の archive は logical document 単位を正とする

## 6.7 duplicate content の扱い

* 同一 `logical_document_id` 内で `content_hash` が一致した場合、新 version は作成しない
* version_no は増やさない
* job / audit に skip 理由を残す

## 6.8 遷移不可時の扱い

* `ready + is_active=false` 以外を approve: `409 conflict`
* `ready` 以外を active にする操作: `409 conflict` 相当のアプリ制御

## 7. jobs 状態遷移

## 7.1 状態一覧
- `queued`
- `running`
- `succeeded`
- `failed`
- `canceled`

### 初期状態
- `queued`

## 7.2 状態遷移表

| 現在状態 | 次状態 | トリガー | 種別 |
|---|---|---|---|
| なし | queued | job 作成 | API/internal |
| queued | running | worker が lease 取得 | job |
| running | succeeded | 正常完了 | job |
| running | failed | 処理失敗 | job |
| queued/running | canceled | 管理操作または内部中止 | internal/admin |
| failed | failed（元 job はそのまま保持） / 新 job を queued 作成 | retry API により新 job 作成 | API |

## 7.3 lease / reclaim ルール
- `running` 中は `locked_by`, `locked_at`, `lease_expires_at` を必須とする
- `lease_expires_at` を過ぎた `running` job は回収候補
- 回収時は新たな worker が処理を引き継げる

## 7.4 retry ルール
- retry は **元 job を戻さない**
- `failed` job に対して **新しい job** を作る
- 新 job は `retry_of_job_id = source job_id`
- `retry_count` は各 job 自身の試行回数
- lineage 全体は `retry_of_job_id` を辿って把握する
- 同一 source job の retry job が `queued/running` 中なら `409 conflict`

## 7.5 遷移不可時の扱い
- `failed` 以外の job を retry: `409 conflict`

## 8.1 状態一覧

* `queued`

* `succeeded`
* `failed`
* `canceled`

### 初期状態

* `queued`

## 8.2 遷移表

| 現在状態           | 次状態       | トリガー                     | 種別             |
| -------------- | --------- | ------------------------ | -------------- |
| なし             | queued    | `POST /evaluations/runs` | API            |
| queued         | running   | evaluator worker 開始      | job            |

## 8.3 ルール

* Phase1 では `trigger_type=manual` のみ許可
* `evaluation_run_items` の集計結果に応じて run を完了させる

## 8.4 `canceled` の Phase1 における位置づけ
- `canceled` は内部中止または将来の管理操作用の予約状態とする
- Phase1 の公開 API では evaluation cancel endpoint は持たない
## 8.5 副作用

* dataset / model / prompt / retrieval settings を run header に固定
* 各 case に対して `evaluation_run_items` を作成

## 10. 横断フロー

## 10.1 temporary chat TTL 到達時の削除フロー

### 対象

* `chat_sessions`
* `chat_messages`
* `chat_tags`
* `summary_memories`
* `retrieval_runs`
* `retrieval_run_items`
* `citations`

### 手順

1. 対象 session の assistant messages の `linked_retrieval_run_id` を `NULL` に更新
2. `retrieval_run_items` / `citations` を削除
3. `retrieval_runs` を削除
4. `summary_memories` を削除
5. `chat_tags` を削除
6. `chat_messages` を削除
7. `chat_sessions` を削除

### 補足

* temporary chat は恒久保存しない
* 関連 trace / citation も保持しない

---

## 10.2 archive 実行時の retrieval 対象外化

### trigger

* `POST /documents/{logical_document_id}/archive`

### 処理

1. `logical_documents.status = 'archived'`
2. 関連する active version を false 化
3. Qdrant payload の `is_active` mirror を更新
4. 以後 retrieval 対象外

### no-op

* すでに archived の場合は `200 OK` + `already_archived`

---

## 10.3 `/rag/ask` replay / duplicate 横断ルール

### 対象キー

* `chat_session_id + client_message_id`

### 分岐

* success replay: 既存結果返却
* in progress: `409 request_in_progress`
* failed replay with same id: `409 conflict`
* same id + different body: `409 conflict`

### 保存ルール

* duplicate request では新 user message を作らない
* failed retry は新 `client_message_id` による新 message 作成で扱う

---

## 10.4 `/rag/search` standalone retrieval run

### 定義
`retrieval_runs` は **chat 起源 run** と **standalone retrieval run** の両方を表す。

### standalone run の条件
- `/rag/search` 起源
- `chat_session_id = NULL`
- `request_message_id = NULL`
- retrieval trace は保存する
- standalone retrieval run でも retrieval trace を構成する `retrieval_run_items` は保存対象とする
- chat history には表示しない

## 11. 実装者向け注意点

### 11.1 Phase1 で DB 制約ではなくアプリ制御に寄せるもの

* same-session 保証
* `linked_retrieval_run_id` が assistant のみであること
* `is_active=true` を ready version のみに制限すること
* `client_message_id` の本文一致 / 不一致判定

### 11.2 DDL で強く支えるもの

* partial unique index
* CHECK 制約
* NULL / NOT NULL
* self-FK
* retry_of_job_id の自己参照禁止

---

## 12. 総括

本書は、Phase1 実装において状態を持つ主語の振る舞いを固定し、API / ER / DDL / 実装の間で再解釈が起きないようにすることを目的とする。

特に以下を重視している。

* temporary chat の削除整合
* edit lineage の再生成整合
* logical document archive と retrieval 対象外化の一貫性
* ready / active version の整合
* job retry / lease / reclaim の明確化
* manual evaluation の状態管理
* `/rag/ask` replay / duplicate と `/rag/search` standalone run の定義固定

以上をもって、状態遷移仕様書 v1.1 とする。

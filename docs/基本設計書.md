## 基本設計書 v1.2

---

## 1. 文書概要

### 1.1 文書名

AI/LLMエンジニア向けポートフォリオ提出用 RAGシステム 基本設計書

### 1.2 目的

本書は、要件定義書で定義した RAG システムについて、実装に必要な主要な構成、責務分離、データ設計方針、処理方式、インターフェース方針を定義する。

本書の目的は以下である。

- 実装者が迷わず開発を開始できること
    
- モジュール境界と責務を明確化すること
    
- Phase1 / Phase2 / Phase3 の拡張余地を保持した設計を示すこと
    
- 要件との対応関係を説明可能にすること
    

### 1.3 設計方針サマリ

- Phase1 は **ローカル検証環境向けの実装成立** を最優先とする
    
- UI / API / CLI / Worker を分離し、保守性と説明性を高める
    
- DB アクセスは Phase1 では **同期 SQLAlchemy 2.x** を採用する
    
- 設計上は将来的な async 化に耐えられる層分離を行う
    
- 会話機能、評価機能、監査、PII 対策を RAG の中核要素として扱う
    
- ローカル実行を基本とし、将来的な外部API切替と AWS 展開に備える
    
- Phase1 は **堅く作り切ること** を優先し、Phase2 以降で observability・評価高度化・モデル改善基盤を強化する

## 4. 技術選定理由

### 4.1 FastAPI + React を採用する理由

- 現在の Web / AI 系システム開発での利用実績と学習コスト、保守性、分離性のバランスがよい
    
- API と UI を明確に分離でき、ポートフォリオとして責務設計を説明しやすい
    
- FastAPI は Pydantic との親和性が高く、schema 駆動で API 設計を実装へ落とし込みやすい
    
- React は状態を持つ UI（chat, admin, evaluation, archived 表示など）との相性が良い
    

### 4.2 SQLAlchemy 2.x + Pydantic v2 + Alembic を採用する理由

- ORM、schema、migration を明確に分離でき、基本設計・ER・API 設計と整合しやすい
    
- 型安全性、保守性、実務説明性が高い
    
- Phase2 以降の拡張でも DB 全面再設計を避けやすい
    

### 4.3 Docker を採用する理由

- Windows 開発 / Ubuntu 運用の差異を吸収しやすい
    
- ローカル検証環境を再現しやすい
    
- Phase1 の検証環境向け deploy と整合する
    

### 4.4 GitHub Actions を採用する理由

- lint / format check / type check / test / build を自動化しやすい
    
- ポートフォリオとして CI/CD の基本導線を示しやすい
    
- Phase1 では検証環境向けの CD、Phase3 以降は AWS への deploy 拡張へつなげやすい
    

### 4.5 uv を採用する理由

- Python の依存管理、仮想環境、実行環境管理を高速かつ一貫して扱える
    
- Windows / Linux をまたぐ開発で依存差異を減らしやすい
    
- Docker / GitHub Actions と組み合わせた再現性の高い開発フローを作りやすい
    
- Phase1 の「まず堅く作る」という方針に合う
    

### 4.6 pytest を採用する理由

- API、service、状態遷移、duplicate / replay などの振る舞いをテストしやすい
    
- 今回のような状態を持つシステムでは、正常系だけでなく競合・失敗・no-op を検証しやすい
    
- Phase1 の品質保証基盤として最も実務的
    

### 4.7 mypy を採用する理由

- SQLAlchemy 2.x + Pydantic v2 + service / repository 分離構成で、型破綻を早期に検出しやすい
    
- request / response schema、service 層、repository 層の責務境界を崩しにくい
    
- 設計通りの責務分離を維持しやすい
    

### 4.8 Ruff を採用する理由

- Python コードの lint / format / import 整理を高速に一元化できる
    
- 開発者ごとの差異や CI 上でのスタイル不整合を減らせる
    
- FastAPI / SQLAlchemy / テストコードを含む Python 中心のコードベースで、日常開発のノイズを大きく減らせる
    
- GitHub Actions と組み合わせることで、品質チェックを軽量かつ継続的に回しやすい
    

### 4.9 LangSmith を Phase2 以降に追加する理由

- Phase1 では retrieval trace / evaluation / confidence / citation を自前設計で見せることを優先する
    
- Phase2 以降で trace 可視化、observability、evaluation の見える化を強化するために追加する
    
- 基盤理解を見せた後に、運用・分析基盤を重ねる流れがポートフォリオとして自然
    

### 4.10 SentenceTransformers を Phase2 以降に追加する理由

- Phase1 では埋め込み・rerank の本番実行を固定モデルで安定させることを優先する
    
- Phase2 以降で fine-tuning、評価、比較実験基盤として活用する
    
- モデル改善フェーズで導入する方が、役割が明確で設計意図を説明しやすい

## 6. レイヤードアーキテクチャ

### 6.1 レイヤ構成

Backend は以下のレイヤに分割する。

1. API Layer
    
2. Schema Layer
    
3. Service Layer
    
4. Repository Layer
    
5. Infrastructure Layer
    

### 6.2 各レイヤの責務

#### API Layer

- FastAPI Router
    
- 入力受付
    
- 認証済みユーザー取得
    
- リクエストバリデーション
    
- レスポンス整形
    

#### Schema Layer

- Pydantic request / response schema
    
- DTO
    
- internal settings schema
    

#### Service Layer

- 業務ロジック
    
- トランザクション単位の処理オーケストレーション
    
- Repository の呼び出し
    
- 外部コンポーネント呼び出し制御
    

#### Repository Layer

- SQLAlchemy を用いた DB アクセス
    
- CRUD
    
- 検索条件組み立て
    
- ORM と永続化境界の管理
    

#### Infrastructure Layer

- File Storage
    
- Qdrant Client
    
- Ollama / LLM Client
    
- OCR Engine
    
- Job 実行基盤
    
- Logging / audit 出力
    

### 6.3 設計原則

- ORM モデルを API レスポンスへ直接返さない
    
- API schema と ORM model を分離する
    
- Service 層に業務ルールを集約する
    
- Repository 層に業務ルールを持ち込まない
    
- 外部依存は interface / adapter で隠蔽する

## 9. 認証・認可設計

### 9.1 認証方式

Phase1 では **HTTP-only Cookie + サーバーサイドセッション** を採用する。

### 9.2 セッション管理

- Cookie にはセッション識別子のみ保持する
    
- セッションは DB で管理する
    
- 失効・ログアウトに対応する

- `viewer`

- 再インデックス
    
- 評価実行
    
- 設定変更
    
- 監査ログ閲覧

#### viewer に許可する主操作

- チャット利用
    
- 会話履歴操作

### 10.3 会話メモリ

会話メモリは以下を組み合わせる。

- 要約メモリ
    
- 直近8件のメッセージ
    

#### ルール

- セッション単位で保持
    
- retrieval の検索対象には含めない
    
- ユーザー意図解釈や文脈補助のみに利用

### 10.4 メッセージ編集

- 編集対象メッセージ以降を再生成対象とする
    
- 旧系列は内部保持する
    
- 明示的な分岐UIは提供しない
    
- 内部的には lineage を追跡可能にする
    

### 10.5 一時チャット

- 通常履歴と分離する
    
- UI離脱後は一覧に表示しない
    
- TTL 120分
    
- TTL 到達後、クリーンアップジョブで物理削除
    
- 将来的な論理削除移行に備え、削除戦略を抽象化する

## 15. 検索・RAG パイプライン設計

### 15.1 採用モデル

- embedding: `BAAI/bge-m3`
    
- reranker: `BAAI/bge-reranker-v2-m3`
    

### 15.2 RAG 処理フロー

1. ユーザー入力受信
    
2. セッション文脈取得
    
3. 要約メモリ + 直近8件参照
    
4. 質問正規化
    
5. retrieval query 生成
    
6. Qdrant 検索
    
7. metadata filtering
    
8. reranking
    
9. context assembly
    
10. answer generation

### 15.3 回答ポリシー

- citation がない主張は禁止
    
- 根拠不足時は拒否または確認
    
- 曖昧質問は暫定解釈を確認
    
- 非公開ソース利用時はその旨を明示
    

### 15.4 旧版検索ルール

- active version 優先
    
- 旧版候補を許可
    
- 採用時明示表示
    

### 15.5 信頼度設計

信頼度は単一値ではなく、少なくとも以下の指標を内部的に保持する。

- retrieval_score_summary
    
- rerank_score_top1
    
- answer_confidence
    
- groundedness_score
    
- confidence_label
    

`confidence_label` は UI 表示用のラベルであり、初期状態では `High / Medium / Low` の3段階を採用する。

### 15.6 信頼度保存と返却

- answer 生成時に confidence 情報を保存する
    
- response schema で返却可能とする
    
- UI は confidence_label と補助説明を表示できること
    
- 閾値は system settings で設定可能とする
    

### 15.7 外部API切替方針

- 外部APIへは retrieval 後の最小コンテキストのみ送る
    
- 文書全文送信は禁止
    
- PII マスキング後のみ送信
    
- API利用履歴は監査ログへ保存

## 17. PII 設計

### 17.1 基本ポリシー

- 原文保存は必要最小限
    
- 暗号化
    
- 厳格 ACL
    
- ログ / 監査 / 評価は不可逆マスクまたは tokenized ID のみ
    
- 外部API送信は不可逆マスク後の最小コンテキスト
    
- 再識別が必要な項目のみ secure mapping store に分離
    

### 17.2 保持期間

- アプリログ: 30〜90日
    
- 監査ログ: 180〜365日
    
- 一時チャット: TTL 削除
    
- 原本: 業務上必要な期間のみ
    

### 17.3 表示制御

- UI 表示は権限に応じてマスク
    
- 明示操作時のみ一部開示
    
- 開示操作は将来的に監査対象化できる構造とする

## 18. データ設計方針

### 18.1 ORM と API の分離

- SQLAlchemy ORM は永続化専用
    
- Pydantic は request / response / internal schema 専用
    
- ORM を API 出力へ直接使わない
    

### 18.2 セッション管理

- background job 単位でも独立した DB session を持つ
    

### 18.3 マイグレーション

### 18.4 ジョブ配送方式

- ポートフォリオとして説明しやすい
    
- 将来 Redis 等へ置換しやすい

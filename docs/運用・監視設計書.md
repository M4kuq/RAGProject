## 2. 運用方針（Phase1）

- ローカル再現性を最優先する
- 監視は「障害の早期発見」と「デモ再現性」に集中する
- 生ログに機密/PII/raw text を残さない
- 複雑な外部監視基盤は必須化しない

---

## 3. ログ設計

## 3.1 application log

必須項目:
- timestamp
- level
- request_id
- actor_user_id（可能時）
- endpoint / job_type
- error_code（失敗時）

禁止項目:
- raw document text
- raw chunk text
- raw prompt
- raw query
- raw LLM response
- secret / token / password

## 3.2 structured log 形式

- JSON line 形式を推奨
- key 名は snake_case
- PII は redacted または hash 化

## 3.3 audit log

- 監査対象操作を audit_logs に保存
- action_type / target_type / target_id / request_id を記録
- target_id NULL 許容ケースを許容

---

## 4. メトリクス設計（軽量）

## 4.1 backend 指標

- request count
- request latency（p50/p95）
- error rate（4xx/5xx）
- `/rag/ask` no_context rate
- citation_build_failed count

## 4.2 worker 指標

- queued / running / failed jobs
- lease lost count
- reclaim count
- retry count
- qdrant_cleanup_failed count

## 4.3 RAG 指標

- retrieval hit rate
- avg selected context count
- confidence label distribution

---

## 5. health / readiness

## 5.1 health check

- backend process alive
- db connection alive
- qdrant reachable（RAG有効時）

## 5.2 readiness

- migration head 適用済み
- 必須 settings 読み込み済み
- worker startup check 成功

---

## 6. failed job / worker 監視

- failed job が閾値超過したら警告
- 同一 job_type の連続失敗を検知
- lease lost の増加を検知
- reclaim が過剰な場合、worker timeout/負荷を疑う

---

## 7. Qdrant 整合チェック

## 7.1 目的

- RDB final check 前提を維持しつつ、Qdrant mirror 乖離を早期検知

## 7.2 チェック項目

- active document_versions に対する point 存在率
- archive 済み文書の point 残存有無
- cleanup 失敗後の再試行必要件数

## 7.3 実行頻度

- Phase1: 手動または日次バッチ（軽量）

---

## 8. backup / restore

## 8.1 対象

- PostgreSQL（必須）
- ファイル保存領域（必須）
- Qdrant（推奨）

## 8.2 方針

- デモ前に DB スナップショット取得
- restore 手順を runbook に明記
- restore 後に migration/version 整合確認

---

## 9. migration 運用

- デプロイ前に `alembic upgrade head`
- ローカル再構築時のみ `downgrade base` 許容
- migration cycle test（upgrade/downgrade/upgrade）を定期実行

---

## 10. retention 方針

- application log: Phase1 は短期保持（例: 7〜14日）
- audit log: デモ期間中は保持
- 大容量 raw データ保持は行わない

---

## 11. alert 候補（Phase1）

- backend 5xx rate 上昇
- `/rag/ask` citation_build_failed 連発
- worker failed job 急増
- DB connection error 連発
- Qdrant 接続不可継続

> Phase1 は通知先を限定（ローカル通知 / Slack 任意）

---

## 12. runbook（最小）

## 12.1 代表インシデント

1. backend 起動不可
2. worker failed job 増加
3. no_context 急増
4. citation_build_failed 増加
5. Qdrant 不整合検知

## 12.2 共通初動

- request_id / job_id で追跡
- 直近 migration 差分確認
- ログで error_code 集計
- 再実行可否を Worker 方針で判断

---

## 13. local / demo 運用手順

- デモ前:
  - migration head 確認
  - seed 適用確認
  - smoke ask/search
  - worker 稼働確認

- デモ後:
  - 失敗 job 有無確認
  - 必要に応じて snapshot へ戻す

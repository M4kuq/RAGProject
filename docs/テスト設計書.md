## 2. テスト戦略

### 2.1 戦略原則

- 同一入力・同一状態で同一結果を検証する（Deterministic）
- DB制約とAPI契約を最優先で検証する
- Worker の lease / reclaim / retry など競合系を重点化する
- raw text / prompt / PII がログへ出ないことを必須確認する

### 2.2 テスト実行順序（推奨）

1. unit / repository
2. service
3. DB constraint / migration
4. API
5. integration（RAG・Worker・Qdrant連携）
6. frontend component / hook
7. E2E
8. security / concurrency 回帰

### 2.3 Phase1 / Phase2 境界

- Phase1: 正常系 + 主要失敗系 + 競合安全性 + セキュリティ基本要件の検証
- Phase2: 高負荷耐性、長時間 soak、運用自動化深堀り

## 3. テストレベル定義

## 3.1 Unit Test

対象:
- 純粋関数、ドメイン判定、error_code mapping、state derivation

必須:
- no_context 判定
- confidence_label 判定
- display_status 導出
- request_id / client_message_id バリデーション

## 3.2 Repository Test

対象:
- SQLAlchemy repository の CRUD / filter / pagination
- composite FK を跨ぐ取得整合

必須:
- retrieval_runs / retrieval_run_items / citations の join 整合
- archived / active / temporary の抽出条件

## 3.3 Service Test

対象:
- transaction boundary
- domain rule enforcement
- idempotency / replay 判定

必須:
- `/rag/ask` replay / duplicate / request_in_progress
- archive 後 retrieval 対象外
- failed ingest retry の状態遷移

## 3.4 API Test

対象:
- request / response schema
- status code
- error response
- permission

必須:
- `/rag/search` 0件時 `200 + items=[]`
- `/rag/ask` no_context_found `422`
- citation_build_failed `500`
- job response は `created_at` を使用（`scheduled_at` 非採用）

## 3.5 Integration Test

対象:
- backend + db + qdrant + worker の連携
- external I/O 分離前提の整合

必須:
- document ingest ～ retrieval ～ citation 生成
- reclaim 後 partial artifact cleanup
- Qdrant mirror update の回復可能性

## 3.6 Frontend Component / Hook Test

対象:
- route guard（admin/viewer）
- CSRF token 更新 single-flight
- TanStack Query cache invalidation

必須:
- loading / empty / error / archived 表示
- optimistic UI と persisted message merge

## 3.7 E2E Test

対象:
- login～chat～upload～ask～citation表示～admin操作

必須:
- viewer と admin の導線差分
- archive 後の UI 挙動
- temporary chat cleanup 後の再表示挙動

## 4. DB Constraint / Migration テスト

### 4.1 DB Constraint Test

重点制約:
- `user_settings` 1:1
- `document_versions` active unique
- `jobs` status / started_at / finished_at CHECK
- `retrieval_runs` standalone run CHECK
- `citations` -> `retrieval_run_items` composite FK
- `chat_messages` <-> `retrieval_runs` 循環参照後付けFK

合格基準:
- 制約違反入力が必ず失敗する
- 正常入力は失敗しない

### 4.2 Migration Test

対象:
- empty DB への full upgrade
- full downgrade
- upgrade -> downgrade -> upgrade の再適用

合格基準:
- migration 順序で循環 FK を破綻させない
- partial unique index が想定どおり作成される

---

## 5. RAG Pipeline テスト

重点:
- no_context / citation_build_failed
- payload_snapshot と CitationItem の分離整合
- failed run で confidence nullable

ケース:
1. `/rag/ask` 正常: retrieval_run + items + citations 保存
2. `/rag/ask` no_context: 422、citation未作成
3. citation_build_failed: 500、原因コード整合
4. `/rag/search` 0件: 200 + items=[]
5. replay: 初回成功と同等 citations を返す

---

## 6. Worker / Job テスト

重点:
- lease renewal / ownership lost / terminal update
- active retry conflict
- message_edit_regeneration
- stale retrieval_run
- evaluation retry / reclaim
- temporary chat cleanup

必須ケース:
1. `FOR UPDATE SKIP LOCKED` による重複取得防止
2. lease lost 時に terminal update しない
3. terminal update 条件 `status='running' AND locked_by=current_worker`
4. failed ingest retry 開始時 cleanup（Qdrant外部、RDB後続）
5. reclaim 後 existing artifact cleanup
6. stale retrieval_run の `request_id + status='running'` 条件 failed 化
7. active retry conflict 拒否
8. evaluation active job 競合時 conflict
9. temporary cleanup 直前 ownership check

---

## 7. Security / Permission テスト

### 7.1 認証認可
- Cookie session 有効期限・失効
- CSRF 必須 endpoint での拒否
- RBAC admin/viewer 境界

### 7.2 入力防御
- upload validation（拡張子・MIME・サイズ）
- path traversal 防止
- request_id / client_message_id フォーマット

### 7.3 ログ非漏洩
- raw document text / raw chunk text / raw prompt / raw query / raw LLM response / PII / secret が application log / audit log に出ない

---

## 8. 失敗系・境界値・並行性テスト

### 8.1 失敗系
- external API timeout / qdrant unavailable / db deadlock retry
- handler exception -> domain update -> runner terminal update

### 8.2 境界値
- page/page_size、token長、タイトル空白、enum外値
- top_k 範囲

### 8.3 並行性
- 同一 chat への同時 ask
- 同一 document_version への retry/reclaim 競合
- 同一 evaluation_run の二重起動競合

## 10. 完了条件（Exit Criteria）

1. 必須テストカテゴリがすべて定義済み
2. 重点テスト観点がケース化済み
3. High severity 未解決 0
4. Medium は修正済みまたは要人間判断として記録済み
5. ログ非漏洩要件の確認ケースを含む
6. Batch 3（migration設計）へ入力可能な粒度

## 11. 停止条件（要人間判断）

- DDL/API/RAG/Worker 間で期待値が一意に定義できない
- 認証認可境界（Cookie/CSRF/RBAC）に仕様衝突
- replay / no_context / citation failure の扱いに矛盾
- 既存確定仕様の変更が必要と判明
